# Cross-Platform Raft Server Runbook

## Scope

This document covers the first runnable server composition in
`cross-platform-raft/src/server/`.

The v1 server composes:

- `FileRaftStorage`
- `RaftCore`
- `RaftRuntime`
- `GrpcRaftTransport`
- `MetadataStateMachine`
- `MetadataService`
- `RaftRpcServiceAdapter`
- `MetadataRpcServiceAdapter`
- `StoreRpcServiceAdapter`
- one `grpc::Server`

## Build

Configure and build with the existing preset only:

```bash
cmake --preset debug-ninja-low-parallel
cmake --build --preset debug-ninja-low-parallel --parallel 2
```

## Run

Start one node with an existing config file:

```bash
./build/linux/cross-platform-raft/src/server/cpr_server_main \
  --config cross-platform-raft/config/node-1.json
```

The process loads config through `ConfigLoader`, initializes the composed
server, starts the runtime, then starts one gRPC server that registers:

- `raft.v1.RaftService`
- `metadata.v1.MetadataService`
- `store.v1.StoreService`

The server keeps running until `ServerLifecycle` receives a shutdown request.

## Endpoints

Endpoints come directly from the config file:

- raft: `ip_address:raft_port`
- metadata: `ip_address:metadata_port`
- store: `ip_address:store_control_port`

If two configured ports are equal, the same underlying gRPC server port is
reused for both services.

## Runtime Notes

- The peer transport pump only forwards outbound Raft messages generated by
  `RaftRuntime`; it does not add retry, pooling, or connection management.
- A fresh single-node bootstrap starts as a local leader so that metadata and
  store RPC paths can run without waiting for a remote vote.
- Metadata writes go through `MetadataService`, then `RaftRuntime`, and only
  return success after the log is committed and applied.

## Metadata RPC Notes

Current metadata RPC behavior is intentionally minimal:

- `Propose` expects `command_payload` to already contain an encoded
  `MetadataCommand`.
- `Query` interprets `query_payload` as the raw target ID bytes.
- `GetLeader` and `GetStatus` read the current runtime snapshot.

These rules match the current adapters and tests. They are not a generic client
SDK contract.

## First-Version Limitations

The current server intentionally does not provide:

- startup recovery from a non-empty persisted Raft state directory
- TLS, authentication, or authorization
- connection pooling, retry, or backoff for peer RPC
- metadata query consistency modes such as `ReadIndex` or lease-read
- production multi-node orchestration, deployment scripts, or system service packaging

If `FileRaftStorage::Load()` finds persisted Raft state, initialization fails
fast with a clear status instead of pretending recovery is implemented.

## Test Constraints

Validation for this server composition currently uses:

```bash
ctest --preset debug-tests -R "server\\." --output-on-failure
ctest --preset debug-tests --output-on-failure
```

Tests use loopback addresses and dynamic ports only. Do not hard-code ports for
parallel local runs.
