# Implementation Plan: Cross-Platform Raft Metadata Foundation

**Branch**: `main` | **Date**: 2026-07-14 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/001-cross-platform-raft/spec.md`

## Summary

Build a reusable, persistent Raft metadata foundation with strict module boundaries: deterministic single-threaded RaftCore, durable RaftLog/RaftStorage, bounded RaftRuntime queues, replaceable transport and state machine boundaries, and a small Metadata control-plane example for stores, placement, and tasks. The implementation proceeds from build/platform foundations through Raft correctness, persistence, concurrency, communication, membership, metadata, example business modules, and final fixtures/tools/acceptance.

No implementation code is generated by this plan. Scope is limited to the completed specification.

## Technical Context

**Language/Version**: C++17  
**Primary Dependencies**: CMake, gRPC, Protocol Buffers, GoogleTest, nlohmann/json, optional spdlog  
**Storage**: File-backed Raft hard state, log records, and snapshots via platform-neutral FileOps; in-memory storage for tests  
**Testing**: GoogleTest unit, integration, recovery, concurrency, and fixture-driven tests  
**Target Platform**: Windows and Linux; MSVC, GCC, and Clang  
**Project Type**: Cross-platform C++ library plus server, RPC services, tools, and tests  
**Performance Goals**: Bounded memory growth under concurrent requests; predictable phase-level correctness before optimization  
**Constraints**: Durability and recovery before responses/messages that depend on persisted state; single-threaded RaftCore; no direct platform APIs outside platform module; no lock-free queues, coroutines, mmap WAL, O_DIRECT, io_uring, or IOCP storage optimization in v1  
**Scale/Scope**: Small configurable Raft metadata clusters with voters and learners, one membership change at a time, and a Metadata example suitable for later replacement

## Overall Architecture

The architecture is a small layered system:

1. `common` and `platform` provide shared types, status handling, bounded queues, checksums, and platform-neutral file/process/time operations.
2. `config` validates JSON into typed runtime settings.
3. `raft` owns consensus data, protocol logic, log, storage boundary, storage implementations, runtime, transport boundary, membership, and snapshots.
4. `metadata` implements the replaceable state-machine example and metadata service behavior through Raft boundaries.
5. `store` implements the Metadata example registry, placement policy, task manager, and task dispatcher.
6. `rpc` adapts gRPC/protobuf traffic into service/runtime queue entries and constructs responses.
7. `server` composes modules and owns lifecycle.
8. `tests` and `tools` validate and inspect the system through public contracts.

Rules:

- `common` and `platform` must not depend on Raft.
- RaftCore may depend only on common/basic Raft types, RaftLog, and protocol data structures.
- RaftStorage, RaftTransport, and RaftRuntime depend on stable Raft interfaces, not business modules.
- Metadata uses Raft through `IRaftStateMachine` and runtime/service boundaries.
- StoreRegistry, PlacementPolicy, and TaskDispatcher belong to the Metadata example layer.
- RPC may call services and runtime queues but must not contain consensus algorithms.
- Server performs composition and lifecycle only.

## Core Interface Plan

No full function bodies are planned here. Interfaces are stable boundaries with clear inputs and outputs.

### RaftCore

- Inputs: logical ticks, RequestVote messages, AppendEntries messages, InstallSnapshot outcomes, local proposals, membership-change requests, persistence-completion notifications.
- Outputs: hard-state changes, log entries to persist, messages to send after persistence, commit-index updates, apply-ready ranges, role/term/status changes.
- Boundary: deterministic, single-threaded, no direct file, network, JSON, platform, thread, time, or business access.

### RaftLog

- Inputs: entries, entry ranges, index/term queries, conflict points, stable/commit/apply/snapshot boundary updates.
- Outputs: append results, conflict locations, truncated suffix state, term lookup result, entry ranges, invariant failures.
- Boundary: enforces `appliedIndex <= commitIndex <= lastIndex` and distinguishes unpersisted, persisted, committed, and applied entries.

### IRaftStorage

- Inputs: storage paths/options, hard state, entry batches, truncation index, snapshot payload and metadata.
- Outputs: loaded hard state/log/snapshot, durable append/truncate/save results, corruption/incomplete-tail diagnostics.
- Boundary: stable storage API; FileRaftStorage uses FileOps, not direct platform APIs.

### IRaftTransport

- Inputs: peer id/address, RequestVote, AppendEntries, InstallSnapshot, cancellation/shutdown events.
- Outputs: delivery result, peer response, transport status.
- Boundary: real gRPC transport and in-memory simulated transport are replaceable behind the same interface.

### RaftRuntime

- Inputs: bounded queues for ticks, proposals, network messages, membership changes, persistence completions, apply completions, and shutdown.
- Outputs: client proposal results, outbound peer work, persistence work, apply work, status snapshots.
- Boundary: owns protocol thread, persistence thread, ordered apply thread, and peer outbound queues; RPC callbacks only enqueue work.

### IRaftStateMachine

- Inputs: committed log index/term and opaque command payload; snapshot payload for restore.
- Outputs: apply result, queryable state update, snapshot payload.
- Boundary: the only Raft-to-business state transition interface.

### IPlacementPolicy

- Inputs: placement request and a snapshot of eligible store registry state.
- Outputs: selected store ids or explicit insufficient-store/capacity result.
- Boundary: default implementation is deterministic simple placement; future policies replace it without Raft changes.

### ITaskDispatcher

- Inputs: store id, poll limit, task state snapshot, task result reports.
- Outputs: polled task list, state updates, duplicate-handling result.
- Boundary: default implementation is FIFO/idempotent; future scheduling replaces it without Raft changes.

## Thread and Queue Model

- gRPC worker threads validate requests, convert protobuf messages, and enqueue work only.
- One Raft protocol thread drains runtime events and drives RaftCore serially.
- One persistence thread serializes durable hard-state, log, truncation, and snapshot work.
- One apply thread applies committed entries in log-index order.
- Each peer has an independent bounded outbound queue.
- All queues have configured capacity. Full queues return BUSY, RESOURCE_EXHAUSTED, RETRY_LATER, or a matching status.
- Shutdown is an explicit runtime event. Runtime stops accepting new work, completes or rejects in-flight work deterministically, joins owned threads, and reports final status.

## Persistence and Recovery Flow

Critical proposal order:

1. Metadata RPC receives a Proposal and validates it.
2. RPC inserts the Proposal into a bounded runtime queue.
3. RaftRuntime delivers the Proposal to the protocol thread.
4. RaftCore appends local log entries conceptually and produces hard-state/log persistence work plus outbound messages that depend on that work.
5. Runtime sends persistence work to the persistence thread.
6. IRaftStorage persists required hard state and log entries, flushes durable data, and reports completion.
7. Runtime releases dependent outbound Raft messages only after required persistence succeeds.
8. Followers persist accepted hard state/log changes before returning successful AppendEntries responses.
9. Leader advances commitIndex only after majority replication rules are satisfied.
10. Apply thread applies committed entries in order.
11. Client success is returned only after the corresponding entry is committed and applied.

Failure rules:

- Success is never returned for an entry that exists only in leader memory or only on the leader local disk.
- Incomplete tails are recovered as incomplete records, not valid entries.
- Checksum mismatch is corruption and must not be ignored.
- Snapshot install/save must preserve snapshot boundary and membership view.
- Windows/Linux flush, truncate, rename, and atomic replacement differences are handled by FileOps.

## gRPC Communication Flow

- `raft.proto`: peer consensus traffic for RequestVote, AppendEntries, and InstallSnapshot.
- `metadata.proto`: client Metadata operations for Propose, Query, GetLeader, GetStatus, AddLearner, PromoteLearner, and RemoveMember.
- `store.proto`: Store control operations for Register, Heartbeat, Stop, Remove, PollTasks, and ReportTaskResult.
- RPC service classes perform validation, protobuf/internal conversion, queue insertion, and response construction.
- RPC service classes do not perform elections, quorum calculations, log conflict handling, or direct RaftCore mutation.
- Non-leader writes return NOT_LEADER and leader address when known.

## Membership-Change Flow

- Bootstrap config creates the initial membership view only.
- AddLearner is proposed and committed through the Raft log.
- Learner receives logs and snapshots but does not vote, count in quorum, or become leader.
- PromoteLearner requires catch-up to leader commitIndex, healthy communication, and no active membership change.
- RemoveMember is committed through the log.
- Only one membership change is active at a time in v1.
- Use an explicit two-phase or equivalent safe transition: enter joint/transition state, replicate and commit it under the old safe rules, then commit the final configuration.
- If the old voter configuration loses quorum, the system does not automatically rewrite membership.

## State-Machine Extension Model

Raft owns ordering, durability gating, snapshots, and membership. Business systems own command meaning and query behavior. A future KV, block-storage, or object-storage system replaces:

- `IRaftStateMachine` implementation.
- Business command encoding/decoding.
- Business RPC/service layer.
- Optional PlacementPolicy and TaskDispatcher if that domain uses them.

RaftCore, RaftLog, RaftStorage, RaftRuntime, RaftTransport, membership management, and fixture framework remain reusable.

## PlacementPolicy and TaskDispatcher Boundaries

- SimplePlacementPolicy uses only StoreRegistry state and deterministic ordering: RUNNING stores, sufficient remaining capacity, no duplicates, remaining capacity descending, store id tie-breaker.
- IPlacementPolicy replacement must not change RaftCore, StoreRegistry durable data, MetadataService primary contract, or RPC contracts.
- SimpleTaskDispatcher supports FIFO polling, poll limit, state updates, result storage, and idempotent duplicate handling.
- ITaskDispatcher replacement must not change RaftCore, MetadataStateMachine, or primary Store RPC contracts.

## Risks and Technical Debt

- The constitution is still a template; repository AGENTS and the specification are the effective gates until a real constitution is ratified.
- Durable flush semantics differ between Windows and Linux; FileOps tests must make platform contracts explicit.
- gRPC integration can blur boundaries; service adapters must stay thin.
- Membership changes are the highest safety risk; keep the first implementation limited to one active transition.
- Snapshot and WAL recovery bugs can corrupt state; recovery tests must be written before relying on integration behavior.
- Optional spdlog must stay optional; logging must not become a required dependency unless explicitly approved.

## Explicitly Excluded First-Version Features

- Real Store data persistence.
- Multi-Raft.
- Cross-Raft-group transactions.
- Advanced placement, automatic data migration, rack awareness, or failure-domain awareness.
- Complex task workflows, dependency scheduling, or priority scheduling.
- Lease Read, Follower Read, or ReadIndex optimization.
- io_uring, O_DIRECT, IOCP storage optimization, mmap-based WAL, or complex WAL segmentation.
- Lock-free queues.
- Coroutines.
- Byzantine fault tolerance.

## Complexity Tracking

No unjustified complexity violations are planned. The stable boundaries are required by the specification because they are replacement points: storage, transport, state machine, placement policy, and task dispatcher.
