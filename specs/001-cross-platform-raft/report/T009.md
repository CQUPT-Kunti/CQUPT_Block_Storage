## RaftLog Public Interface

`RaftLog` exposes one compact class with these public capabilities:

- Append:
  - `Append(const LogEntry&)`
  - `Append(const std::vector<LogEntry>&)`
- Query:
  - `GetEntry(index, out)`
  - `GetTerm(index, out)`
  - `GetEntries(begin, end, out)`
  - `first_index()`
  - `last_index()`
  - `empty()`
  - `size()`
- Match / conflict:
  - `Matches(index, term)`
  - `FindConflict(incoming, out)`
  - `TruncateSuffix(index)`
- Progress markers:
  - `AdvanceStableTo(index)`
  - `AdvanceCommitTo(index)`
  - `AdvanceAppliedTo(index)`
- Snapshot boundary:
  - `UpdateSnapshotBoundary(index, term)`
  - `snapshot_index()`
  - `snapshot_term()`
- Marker getters:
  - `stable_index()`
  - `commit_index()`
  - `applied_index()`

## Internal Container And Range Semantics

- Internal log storage uses a single `std::vector<LogEntry>`.
- The vector stores only entries strictly after the snapshot boundary.
- Range reads use left-closed right-open semantics: `[begin, end)`.
- Empty log semantics:
  - `first_index() == snapshot_index + 1`
  - `last_index() == snapshot_index`
  - For a fresh log with no snapshot: `first_index() == 1`, `last_index() == 0`

## Append Behavior

- Single append is delegated to batch append.
- Batch append requires:
  - append start index equals `last_index() + 1`
  - indexes remain strictly contiguous
  - indexes stay above snapshot boundary
  - terms are positive
- Duplicate indexes, skipped indexes, and wrong append start all return explicit `InvalidArgument`.
- Append does not perform persistence.

## Query And Match Behavior

- `GetEntry` returns a copied `LogEntry`, so no mutable internal reference is exposed.
- `GetTerm` returns:
  - `0` for index `0`
  - snapshot term for `snapshot_index`
  - normal entry term for in-range log entries
- Access before or at snapshot boundary as a normal entry is rejected.
- `Matches(index, term)` returns true only when both index and term match.
- Empty initial state supports `Matches(0, 0)`.

## Conflict Detection And Truncation

- `FindConflict` returns:
  - `NONE` when incoming entries are fully consistent
  - `CONFLICT` at the first same-index/different-term position
  - `APPEND` at the first incoming index beyond local tail
- Incoming entries must already be contiguous and ordered.
- `TruncateSuffix(index)` removes entries with index `>= index`.
- Truncation rejects:
  - snapshot boundary or earlier
  - committed entries
  - indexes beyond append position
- After truncation, `stableIndex` is clamped so it never points beyond `lastIndex`.

## Marker Update Rules

- `stableIndex`, `commitIndex`, and `appliedIndex` are monotonic.
- Rejected cases:
  - backward movement
  - `stableIndex > lastIndex`
  - `commitIndex > lastIndex`
  - `appliedIndex > commitIndex`
- Failed updates return explicit `InvalidArgument`.
- Object state is updated only after the requested transition passes validation.

## Snapshot Boundary Handling

- Snapshot boundary tracks:
  - `last included index`
  - `last included term`
- Boundary affects:
  - `first_index()`
  - `last_index()`
  - `GetTerm()`
  - `Matches()`
- When the boundary advances, already covered in-memory log prefix entries are erased.
- Boundary cannot move backward.
- Boundary cannot exceed `lastIndex`.
- Boundary term must match the current log term at that index.

Safety note:

- The task text said the new snapshot boundary must not be smaller than the applied index.
- This implementation used the stricter safety rule `snapshot_index <= applied_index`.
- Reason: allowing the snapshot boundary to move beyond applied entries would discard unapplied log state in memory.

## Maintained Invariants

- `appliedIndex <= commitIndex <= lastIndex`
- `stableIndex <= lastIndex`
- `snapshotIndex <= lastIndex`
- After snapshot advancement:
  - `snapshotIndex <= stableIndex`
  - `snapshotIndex <= appliedIndex`
  - `snapshotIndex <= commitIndex`
- Stored log entries remain contiguous from `snapshotIndex + 1`
