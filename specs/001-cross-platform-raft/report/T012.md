## RequestVote Rules

Implemented request handling:

- stale term request is rejected
- higher term request updates current term and forces non-learner nodes back to `Follower`
- candidate log freshness is compared by:
  - last log term first
  - last log index second when terms are equal
- vote is granted only when:
  - current node is not a learner
  - local node has not voted in the term, or already voted for the same candidate
  - candidate log is at least as up to date as local log
- successful vote records `voted_for`
- successful vote resets election ticks
- same-term second vote to a different candidate is rejected
- repeated request from the same candidate in the same term returns the same grant decision

Implemented response handling:

- higher term response makes the current node step down
- stale response does not change term or role
- same-term candidate responses return only minimal action:
  - `RECORDED_GRANT`
  - `RECORDED_REJECTION`
- no majority counting or automatic leadership transition is implemented

`RequestVoteResponse` was minimally extended with `VoteRejectReason` so rejection state is explicit.

## Log Freshness Comparison

`RaftCore` uses `RaftLog` public queries only:

- `last_index()`
- `GetTerm(last_index(), ...)`

Comparison rule:

1. higher last log term wins
2. if last log terms are equal, higher or equal last log index wins

No log freshness logic is duplicated outside `RaftLog` accessors.

## Heartbeat Handling

Heartbeat is implemented as empty-entry `AppendEntries`.

Implemented behavior:

- `BuildHeartbeat()` builds a minimal request from current leader state
- request contains:
  - current term
  - leader id
  - current `prev_log_index`
  - current `prev_log_term`
  - empty `entries`
  - current `leader_commit`
- valid heartbeat records current leader id
- valid heartbeat resets election ticks
- stale heartbeat is rejected
- higher term heartbeat updates term and makes candidate step down to follower
- learner accepts heartbeat as a learner and does not start elections

No timer scheduling, peer queue, or network send logic was added.

## AppendEntries Handling

Implemented request validation and in-memory log update flow:

- stale term request is rejected
- higher term request updates current term
- same-term candidate receiving legal leader request becomes follower
- leader id is recorded on accepted leader traffic
- election ticks reset on accepted leader traffic
- `prev_log_index` and `prev_log_term` are checked through `RaftLog::Matches()`
- mismatch returns failure plus conflict metadata
- matching requests use `RaftLog::FindConflict()` to locate:
  - no-op duplicate ranges
  - first conflicting index
  - first append position beyond the local tail
- conflicting uncommitted suffix is truncated with `RaftLog::TruncateSuffix()`
- new continuous suffix is appended with `RaftLog::Append()`
- leader commit advances local commit index only to:
  - `min(request.leader_commit, log_.last_index())`

Protection rules preserved:

- committed entries are not truncated
- snapshot boundary is not bypassed
- any invalid log mutation returns explicit `Status`
- no persistence, file access, or durable-success meaning was added

Implemented response handling:

- higher term response makes the current leader step down
- stale response is ignored
- same-term leader response returns minimal action:
  - `ACCEPTED`
  - `REJECTED`
- no `nextIndex`, `matchIndex`, catch-up loop, or majority commit logic was added

## Stale And Higher Term Rules

- stale `RequestVote`: reject, keep current state
- higher term `RequestVote`: update term, clear old vote, re-evaluate vote request
- stale `AppendEntries`: reject, keep current state
- higher term `AppendEntries`: update term, accept leader authority if log checks pass
- stale responses: ignored
- higher term responses: step down and clear leadership state
