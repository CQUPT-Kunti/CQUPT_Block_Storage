## PeerProgress Initialization And Update Rules

`RaftCore::Options` now accepts fixed member sets:

- `voter_ids`
- `learner_ids`

Validation rules:

- at least one voter is required
- voter and learner ids must be valid
- voter and learner ids must be unique across both sets
- current node must exist in exactly one set
- learner node must initialize as `LEARNER`
- voter node cannot initialize as `LEARNER`
- learner cannot initialize as `LEADER`

Peer progress storage:

- `RaftCore` stores one `PeerProgress` per configured member
- current node is included in the same map
- current node progress always reflects local `RaftLog`

Initialization behavior:

- all peers start with `nextIndex = lastIndex + 1`
- local node starts with `matchIndex = lastIndex`
- remote peers start with `matchIndex = 0`
- learner flag comes from the fixed member set
- on `BecomeLeader()`, all peer progress is reinitialized from the current local log tail

Update behavior:

- successful replication only moves `matchIndex` and `nextIndex` forward
- repeated or stale success responses do not move progress backward
- failed responses only back off `nextIndex`
- learner progress is updated the same way as voter progress
- learner progress is never counted in majority commit

## AppendEntries Build Strategy

Added public interface:

- `BuildAppendEntriesForPeer(node_id, request)`

Build rules:

- only leader can build peer-specific append requests
- self node cannot be used as remote target
- `prevLogIndex = nextIndex - 1`
- `prevLogTerm` is read through `RaftLog::GetTerm()`
- entries are read through `RaftLog::GetEntries(nextIndex, lastIndex + 1, ...)`
- `leader_commit` is the current local `commitIndex`
- if `nextIndex == lastIndex + 1`, the result is an empty-entry heartbeat

Snapshot boundary behavior:

- if peer `nextIndex` is at or before the snapshot boundary, normal log entries cannot be built
- current implementation returns `RETRY_LATER` with explicit snapshot-needed status
- no `InstallSnapshot` request is built in T013

## Successful And Failed AppendEntries Responses

Successful response handling:

- stale term response is ignored
- higher term response follows T012 and steps leader down
- same-term success updates:
  - peer `matchIndex`
  - peer `nextIndex = max(old_next, matchIndex + 1)`
- success never lets progress move backward
- current leader then tries to advance `commitIndex`

Failed response handling:

- stale term failure is ignored
- higher term failure still steps leader down
- same-term failure uses:
  - `conflictTerm` when available for fast backoff
  - otherwise `conflictIndex`
- backoff is single-step per response
- no retry loop, network send, or local log mutation is performed

## `conflictIndex` And `conflictTerm` Usage

Conflict backoff rule:

1. if follower returns `conflictTerm` and leader has that term locally:
   - find the last local index with that term
   - set `nextIndex = last_local_index_of_conflict_term + 1`
2. otherwise:
   - set `nextIndex = conflictIndex`
3. clamp `nextIndex` into a legal range:
   - lower bound: snapshot boundary sentinel or first normal index
   - upper bound: `lastIndex + 1`

This allows fast catch-up when the leader still has the conflicting term locally, without implementing any resend loop.

## Voter And Learner Handling

- both voters and learners have `PeerProgress`
- both voters and learners can receive replication progress updates
- only voters participate in majority commit
- learner count does not change quorum size
- learner cannot initialize as leader
- learner cannot become leader through the existing transition API

## Majority Calculation

Leader majority commit uses voter `matchIndex` values only:

- collect `matchIndex` for all voters, including leader itself
- sort the values
- choose the middle element as the majority candidate index
- only advance commit when:
  - candidate index is greater than current commit
  - candidate index term equals current leader term

Examples now supported:

- 3 voters require 2 replicated copies
- 5 voters require 3 replicated copies
- learners do not affect the threshold

## Current-Term Commit Restriction

Direct leader commit advancement is restricted to current-term entries only:

- old-term entries replicated to a majority are not directly committed by this T013 logic
- once a current-term entry reaches majority, leader can move `commitIndex` forward through that point

`HardState.commit_index` is updated in memory whenever `RaftLog` commit index moves.

## Apply-Ready Range Interface

Added public interface:

- `GetApplyReadyRange(ApplyRange*)`

Range semantics:

- `begin = appliedIndex + 1`
- `end = commitIndex`
- if `begin > end`, the range is empty
- no state machine call is performed
- `appliedIndex` is not advanced here
