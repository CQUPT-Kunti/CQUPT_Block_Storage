## Output Batch Structure And Public Interface

Added `RaftCore::RaftOutput` and two public interfaces:

- `GetOutput(RaftOutput *output)`
- `ConfirmPersistence(bool hard_state_persisted, LogIndex stable_index)`

`RaftOutput` contains:

- `has_hard_state`
- `hard_state`
- `unstable_entries`
- `immediate_messages`
- `persisted_messages`
- `committed_range`
- `role`
- `term`
- `leader_id`

Output reads are non-mutating:

- repeated `GetOutput()` does not change term, role, log state, or peer progress
- pending output is not silently dropped before explicit persistence confirmation

## HardState Change Detection

`RaftCore` now tracks:

- current in-memory `hard_state_`
- last confirmed `persisted_hard_state_`

`has_hard_state` becomes true when any of these fields differ from the persisted copy:

- `current_term`
- `voted_for`
- `commit_index`

No redundant HardState output is produced when the persisted and in-memory values are identical.

## Unstable Log Range Calculation

Unstable log output is derived from `RaftLog` state:

- begin: `stableIndex + 1`
- end: `lastIndex`

Rules:

- range is continuous
- already stable entries are not emitted again
- snapshot boundary and earlier indexes are never exposed as normal unstable log entries
- `ConfirmPersistence(..., stable_index)` is the only T014 path that advances `stableIndex`

## Immediate And Persisted Message Classification

`RaftCore` now stages outbound messages into two buckets:

- `immediate_messages`
- `persisted_messages`

Classification rules:

- messages that do not depend on newly changed HardState or unstable log entries go to `immediate_messages`
- messages that depend on term/vote/commit persistence or new log entry persistence stay in `persisted_messages`
- `ConfirmPersistence()` reclassifies blocked messages once dependencies are satisfied

No transport queue, runtime scheduler, callback chain, or async worker was added.

## RequestVote And AppendEntries Persistence Gating

RequestVote response gating:

- granted vote responses wait for corresponding `term` / `voted_for` HardState persistence
- stale-term rejections that do not change persistent state are immediately sendable
- repeated reads keep the same classification until persistence is confirmed

AppendEntries response gating:

- successful append responses wait for accepted log entries to become stable
- if the append also changes `term` or `commit_index`, the response also waits for HardState persistence
- mismatch or stale rejections that do not create new persistent state can be immediate

Leader append request gating:

- `BuildAppendEntriesForPeer()` now stages the built request as output
- requests carrying newly appended local log entries are delayed until those entries are stable
- empty heartbeat-style requests remain immediate unless there is still dirty HardState to persist

## Committed Range Output

Committed-but-not-applied output reuses the existing T013 apply-ready logic:

- begin: `appliedIndex + 1`
- end: `commitIndex`
- empty when `begin > end`

No state machine call, apply thread, or `appliedIndex` mutation was added.
