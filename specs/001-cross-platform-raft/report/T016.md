## IRaftStorage Public Interface

`IRaftStorage` defines one minimal synchronous storage boundary:

- `Open(const std::filesystem::path &storage_path)`
- `Load(RaftStorageLoadResult *result) const`
- `SaveHardState(const HardState &hard_state)`
- `AppendEntries(const std::vector<LogEntry> &entries)`
- `TruncateSuffix(LogIndex first_index)`
- `SaveSnapshot(const SnapshotData &snapshot)`

The interface uses:

- `HardState`
- `LogEntry`
- `SnapshotMetadata`
- `OpaquePayload`
- project `Status`
- C++17 standard library types

It does not expose file descriptors, platform handles, FileOps, Config, JSON, gRPC, Protobuf, runtime, transport, Metadata, Store, or Server types.

## Load Result Structure

Added `RaftStorageLoadResult`:

- `hard_state`
- `entries`
- `snapshot`
- `empty`

Snapshot is represented with `std::optional<SnapshotData>`, not a raw pointer. `SnapshotData` contains:

- `SnapshotMetadata metadata`
- `OpaquePayload payload`

`Load()` returns copies of in-memory state and does not expose mutable internal containers.

## MemoryRaftStorage Internal State

`MemoryRaftStorage` maintains:

- `open_`
- current `HardState`
- continuous `std::vector<LogEntry>`
- optional `SnapshotData`

All operations are memory-only. It does not access files, call FileOps, flush, sleep, read time, create threads, or simulate disk latency.

## Open Behavior

- First `Open()` marks the storage as open.
- Repeated `Open()` is idempotent and does not reset stored state.
- The path parameter is accepted for interface compatibility with future file-backed storage and ignored by the memory implementation.
- Operations other than `Open()` return explicit `InvalidArgument` when called before open.

## HardState Save And Validation

`SaveHardState()` validates before assignment:

- `current_term` cannot move backward.
- `commit_index` cannot move backward.
- `commit_index` cannot exceed the latest available log index or snapshot boundary.

On failure, the previous `HardState` remains unchanged. `Load()` returns the last successfully saved `HardState`.

## Log Append And Truncate Rules

`AppendEntries()` behavior:

- empty batch succeeds without mutation
- batch indexes must be contiguous
- terms must be positive
- first new entry cannot skip indexes
- entries at or before the snapshot boundary are rejected
- fully identical repeated entries are idempotent
- overlapping different entries are rejected instead of silently overwritten
- validation happens before appending any suffix

`TruncateSuffix(first_index)` behavior:

- removes entries with index `>= first_index`
- keeps entries before `first_index`
- rejects truncation at or before snapshot boundary
- rejects truncation of committed entries
- rejects indexes beyond the append position
- failed truncation leaves the log unchanged

## Snapshot Save And Cleanup

`SaveSnapshot()` validates before mutation:

- snapshot index must be positive
- snapshot term must be positive
- snapshot boundary cannot move backward
- snapshot index cannot exceed committed index
- snapshot term must match the stored term at the snapshot index

On success:

- latest snapshot metadata and payload are stored in memory
- log entries covered by the snapshot are removed
- remaining log entries keep contiguous indexes after the snapshot boundary

No file format, checksum, temporary file, fsync, or atomic replacement behavior was implemented.

## Data Consistency And Atomic Update Rules

Maintained invariants:

- log indexes remain continuous after the snapshot boundary
- HardState term and commit index are monotonic
- commit index never exceeds the latest available log or snapshot index
- snapshot boundary never moves backward
- committed log entries cannot be truncated
- failed operations validate first and do not leave partial updates
