## File Layout

- Storage root: caller-provided `std::filesystem::path`.
- `hard_state.bin`: one HardState record.
- `log.bin`: ordered LogEntry records.
- T017 does not create a snapshot file or snapshot directory.
- Missing storage directories are created through `platform::CreateDirectories`.
- Repeated `Open()` is safe: it recreates missing files, reloads both files, and replaces the in-memory cache with loaded state.

## Record Format

Each record is encoded with fixed-width little-endian fields, never by writing C++ struct memory:

```text
uint32 format_version
uint32 record_type
uint64 index
uint64 term
uint64 payload_length
uint64 checksum
byte[payload_length] payload
```

- `format_version`: currently `1`.
- `record_type`: `1` for HardState, `2` for LogEntry.
- `checksum`: project `common::Checksum` over `format_version`, `record_type`, `index`, `term`, `payload_length`, and payload, excluding the checksum field itself.
- Payload length limit: 64 MiB.
- Decode checks version, record type, payload length, complete tail, and checksum.

HardState payload:

```text
uint64 voted_for
uint64 commit_index
uint64 membership_configuration_id
```

LogEntry payload:

```text
uint32 entry_type
byte[] opaque_payload
```

## HardState Persistence

- `SaveHardState()` validates that term and commit index do not move backward.
- Commit index must not exceed the last available log index.
- The HardState file is written through `platform::AtomicReplace` in the same directory.
- `FileOps` performs flush before replacement; a write/flush/replace failure is returned as failure and the cached state is not advanced.
- `Load()` returns the last successfully saved HardState and reports corruption for malformed or checksum-invalid content.

## Log Persistence

- `AppendEntries()` accepts empty batches.
- Non-empty batches must be contiguous and may not skip indexes.
- Existing identical entries are treated as idempotent.
- Existing different entries are rejected; there is no silent overwrite.
- The log file is regenerated through `platform::AtomicReplace` after validation, so failed appends do not partially update the in-memory cache.
- `Load()` restores entries in order and rejects non-contiguous records.
- Checksum mismatch returns `Status::Corruption`.
- Incomplete record headers or payload tails return `Status::Corruption`; T017 does not attempt tail repair.

## TruncateSuffix

- `TruncateSuffix(first_index)` keeps entries before `first_index`.
- It rejects truncation of committed entries.
- It rejects indexes beyond the append position.
- A no-op truncate at `last_index + 1` succeeds.
- The truncated log is rewritten through `platform::AtomicReplace`; reload sees the same truncated result.
