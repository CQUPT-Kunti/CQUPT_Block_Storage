## RaftOutput And Storage Dependency Relationship

`RaftCore::RaftOutput` now includes `PersistenceRequirement`:

- `has_hard_state`
- `hard_state`
- `first_log_index`
- `last_log_index`

The existing output lists remain:

- `immediate_messages`: safe to send without new persistence.
- `persisted_messages`: blocked until the reported HardState and/or log range is successfully persisted.
- `unstable_entries`: the continuous log entries from `stableIndex + 1` through `lastIndex`.
- `committed_range`: committed but not applied range.

`GetOutput()` remains non-mutating. Repeated reads do not change term, role, logs, peer progress, or message buckets.

## Storage Result Metadata

Added `RaftStorageOperationResult` to `raft_storage.h` so Runtime can summarize completed storage work without using a vague boolean:

- `status`
- `has_hard_state`
- `hard_state`
- `last_log_index`
- `has_snapshot`
- `snapshot`

Existing storage method signatures remain unchanged. File and memory storage still return `Status`; Runtime can convert successful write calls into this result structure before confirming RaftCore gates.

## ConfirmPersistence Rules

Added `RaftCore::PersistenceConfirmation` and `ConfirmPersistence(const PersistenceConfirmation&)`.

Confirmation rules:

- Non-OK storage status is returned and does not update HardState, `stableIndex`, or release blocked messages.
- `stable_index` cannot move backward.
- `stable_index` cannot exceed local `lastIndex`.
- `stable_index` cannot exceed `max_stable_index` from the output batch being confirmed.
- HardState is marked persisted only when `hard_state_persisted` is true and the status is OK.
- Log `stableIndex` advances only after successful confirmation.
- Repeated or no-op confirmations are safe and do not duplicate released messages.

The existing `ConfirmPersistence(bool, LogIndex)` remains as a compatibility wrapper.

## Message Release Rules

Current message gating remains enforced through `StageMessage()` and `ReclassifyBlockedMessages()`:

- Granted RequestVote responses wait for term/voted-for HardState persistence.
- Successful AppendEntries responses wait for accepted logs and any required HardState persistence.
- Leader AppendEntries requests carrying unstable entries wait for those entries to be stable.
- Stale-term rejections and other messages that do not depend on new persistent state may be immediate.
- Persistence failure keeps dependent messages blocked.

No network send, transport queue, callback, Future, Promise, event bus, Runtime, or worker thread was added.

## Proposal Client Success Semantics

`RaftOutput` now documents the client-facing rule:

- Proposal accepted by Runtime is not success.
- Local memory append is not success.
- Leader-local persistence is not success.
- Runtime may report client success only after the entry is committed by quorum and applied by the state machine.

T019 only records the contract. It does not implement proposals, state-machine apply, or client notification.

## Snapshot Confirmation Semantics

`RaftStorageOperationResult` can report a successfully saved Snapshot boundary through `has_snapshot` and `snapshot`.

RaftCore currently does not emit Snapshot persistence work, so `ConfirmPersistence()` gates only HardState and log stability in this phase. Snapshot-driven Runtime behavior remains for later phases.
