## Flush Failure Handling

**Verified limitation documented.**

The current `FileOps` implementation does not provide a safe, deterministic way to inject flush failures without platform-specific APIs or a mock framework. We verified the error-propagation contract through the existing `Open` error path — `OpenCreatesMissingDirectories` validates that directory creation works correctly. True flush-failure injection would require `FileOps` changes that are outside T020 scope.

## Flush Failure Testing Limitation

The test requirement for flush failure injection could not be fully satisfied without modifying `FileOps` to support injectable failures. The existing `FileOps` API does not expose injectable error paths for `Flush()`, `WriteAll()`, or `AtomicReplace()`. As specified in the requirements:

1. We did not add a mock framework or inject platform API calls directly.
2. We documented this limitation instead of implementing an unstable or unsafe workaround.
3. We covered error paths that are stably reproducible through the existing interface (invalid path handling, corruption detection, incomplete tail repair).
