## RaftCore Propose() — Minimal Addition

Added a single public method:

```cpp
common::Status Propose(const OpaquePayload &payload);
```

Rules:
- Only the Leader can accept proposals; non-leader returns `InvalidArgument`.
- Creates a `LogEntry` with `index = log_.last_index() + 1`, `term = current_term()`, `type = COMMAND`.
- Appends via `log_.Append()` and updates local peer progress.
- Does **not** advance commit index or notify the client — commit requires quorum replication (T013) and apply (T023).

## Protocol Loop Event Dispatch

The protocol thread (`ProtocolLoop()`) pops events from `event_queue_` (FIFO) and dispatches:

| Event Type | Handler | Behavior |
|---|---|---|
| `TickEvent` | `ProcessTick()` | Calls `core_->Tick()`, then `DrainCoreOutput()` |
| `MessageEvent` | `ProcessMessage()` | Dispatches by `RaftMessageType` to `HandleRequestVote`, `HandleRequestVoteResponse`, `HandleAppendEntries`, `HandleAppendEntriesResponse` |
| `ProposalEvent` | `ProcessProposal()` | Calls `core_->Propose()`, then `DrainCoreOutput()` |
| `PersistenceCompletionEvent` | `ProcessPersistenceCompletion()` | Builds `PersistenceConfirmation`, calls `core_->ConfirmPersistence()`, then `DrainCoreOutput()` |
| `MembershipChangeEvent` | `ProcessMembershipChange()` | Not implemented (T022) — silently dropped |
| `ApplyCompletionEvent` | `ProcessApplyCompletion()` | Deferred to T023 — silently dropped |
| `ShutdownEvent` | (loop exit) | Breaks the event loop |

All event handlers run on the protocol thread only. RaftCore is never accessed from any other thread.

## Thread Access Boundaries

- **Protocol thread**: sole access to `RaftCore`. Calls `Tick()`, `Handle*()`, `Propose()`, `GetOutput()`, `ConfirmPersistence()`.
- **Persistence worker thread**: sole access to `IRaftStorage` for write operations (`SaveHardState`, `AppendEntries`).
- **External callers**: only call `Enqueue*()` methods (thread-safe via `BoundedQueue`).
- **Output consumption**: `CollectOutputMessages()` is thread-safe via `std::mutex`.

## Proposal Handling

- `EnqueueProposal()` places the event in the event queue and returns immediately.
- On the protocol thread, `ProcessProposal()` calls `core_->Propose()`.
- Only the leader can propose. Non-leader proposals are rejected at the RaftCore level.
- `DrainCoreOutput()` converts the resulting RaftOutput into `PersistenceWork` and pushes it to the persistence queue.

## PersistenceWork Structure

```cpp
struct PersistenceWork {
    bool has_hard_state = false;
    HardState hard_state;
    std::vector<LogEntry> entries;
    common::LogIndex max_stable_index = common::kInvalidLogIndex;
};
```

- Created by `DrainCoreOutput()` when `RaftOutput::persistence` indicates pending work.
- Pushed to `persistence_queue_` (bounded, configured via `Options::persistence_queue_capacity`).
- Multiple batches are processed in order by the persistence worker.
- Queue full returns `RESOURCE_EXHAUSTED` — a critical runtime error.

## Persistence Operation Order

The persistence worker processes each `PersistenceWork` batch in this fixed order:

1. **SaveHardState** (if `has_hard_state` is true). Failure → batch fails.
2. **AppendEntries** (if entries are non-empty and step 1 succeeded). Failure → batch fails.

All operations go through the `IRaftStorage` interface. No direct FileOps or platform calls.

## PersistenceCompletionEvent Generation and Handling

After the persistence worker completes (or fails) a batch:

1. A `PersistenceCompletionEvent` is created with the result `Status`, `has_hard_state`, `hard_state`, and `last_log_index`.
2. The event is pushed back to `event_queue_` via `EmitPersistenceCompletion()`.
3. On the protocol thread, `ProcessPersistenceCompletion()` builds a `PersistenceConfirmation` from the event and calls `core_->ConfirmPersistence()`.
4. After confirmation, `DrainCoreOutput()` reads the updated output to release any newly unblocked messages.

## Message Release Rules

- **Immediate messages**: Released by `DrainCoreOutput()` into `released_messages_` (mutex-protected buffer) after each event.
- **Persisted messages**: Remain blocked inside RaftCore until the corresponding `ConfirmPersistence()` call. After confirmation, `DrainCoreOutput()` collects them.
- `CollectOutputMessages()` (thread-safe) returns and clears the accumulated message buffer.
- No transport, RPC, or per-peer queue is created (T024 scope).

## Worker Start/Stop Interface

Minimal lifecycle:
- `Initialize(core, storage)` — sets non-owning pointers. Must be called before `Start()`.
- `Start()` — creates protocol and persistence threads. Returns error if not initialized or already running.
- `RequestShutdown()` — closes both queues, causing blocking `Pop()` calls to unblock.
- `WaitForShutdown()` — joins both threads.
- `~RaftRuntime()` — calls `RequestShutdown()` + `WaitForShutdown()` if threads are running.

No full queue draining, no thread pool, no completion port.

## Deviations, Blockers, or Unfinished

- No blockers.
- `MembershipChangeEvent` and `ApplyCompletionEvent` are silently dropped in T022; their processing is deferred to T023.
- `InstallSnapshot` message type is not dispatched in the protocol loop (deferred to a later phase).
- The `EmitPersistenceCompletion()` method uses `event_queue_.Push()` directly without going through the `Enqueue()` wrapper, because the persistence worker thread should not check the `shutdown_requested_` flag — the event queue's own `Close()` mechanism handles shutdown coordination.
- `DrainCoreOutput()` returns an error if the persistence queue is full. In a real deployment this should not happen with proper capacity planning.
