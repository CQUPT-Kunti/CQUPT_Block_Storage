## Apply Interface

```cpp
using ApplyFn = std::function<common::Status(const LogEntry &entry)>;
```

- Injected via `Initialize(core, storage, apply_fn)`. Optional — if null, apply reports success for all entries.
- Called by the apply worker thread only.
- Input is the existing `LogEntry` with opaque payload — no business dependencies.

## ApplyWork and Apply Queue

```cpp
struct ApplyWork {
    common::LogIndex index;
    common::Term term;
    LogEntryType type;
    OpaquePayload payload;
    std::uint64_t proposal_id;  // 0 = no associated proposal
};
```

- Pushed to `common::BoundedQueue<ApplyWork>` with capacity configured via `Options::apply_queue_capacity`.
- Entries are pushed strictly in LogIndex order by `ScheduleApplyWork()`.
- No duplicate indexes are pushed — `next_apply_schedule_` tracks the last scheduled index.
- Queue full returns `RESOURCE_EXHAUSTED`.

## Apply Worker Order Guarantees

- The apply worker thread (`ApplyLoop()`) pops work FIFO from `apply_queue_`.
- Each entry is applied one at a time — no parallel apply.
- After each apply (success or failure), an `ApplyCompletionEvent` is pushed back to the event queue.
- The protocol thread processes completions sequentially, so only one apply is in-flight at a time.

## Committed Entry Scheduling

`ScheduleApplyWork()` (called from `DrainCoreOutput()`):
1. Reads `GetApplyReadyRange()` from RaftCore.
2. Starts from `max(applied_index + 1, next_apply_schedule_ + 1)`.
3. For each index, calls `log_.GetEntry()` to fetch the `LogEntry`.
4. Looks up the associated `proposal_id` from `pending_proposals_`.
5. Creates `ApplyWork` and pushes to `apply_queue_`.
6. Advances `next_apply_schedule_`.

Only committed entries (within the `committed_range`) are scheduled. Uncommitted entries remain in the log but are not applied.

## ApplyCompletionEvent Handling

`ProcessApplyCompletion(const ApplyCompletionEvent &event)`:

| Condition | Action |
|-----------|--------|
| `event.status` is not OK | Produce failure `ProposalResult`, do **not** advance `appliedIndex` |
| `event.applied_index != expected` | Out-of-order or duplicate — reject silently |
| `ConfirmApplied(event.applied_index)` fails | Reject silently |
| All checks pass | Advance `appliedIndex`, produce success `ProposalResult` with associated proposal_id |

## appliedIndex Advancement and Duplicate Protection

- `RaftCore::ConfirmApplied(LogIndex index)` validates:
  - `index == log_.applied_index() + 1` (strictly sequential)
  - `index <= log_.commit_index()` (cannot exceed committed range)
- Calls `RaftLog::AdvanceAppliedTo(index)`.
- Duplicate completions for the same index are rejected because `index != applied_index + 1`.
- Out-of-order completions are rejected for the same reason.

## Proposal ID to Log Index Correlation

- When `ProcessProposal()` succeeds on a leader:
  - Records `pending_proposals_[proposal_id] = core_->log().last_index()`.
- When `ScheduleApplyWork()` creates `ApplyWork`:
  - Searches `pending_proposals_` for a matching log index.
- When `ProcessApplyCompletion()` produces a result:
  - Associates the result with the original `proposal_id`.
  - Erases the entry from `pending_proposals_`.

## ProposalResult Output

```cpp
struct ProposalResult {
    std::uint64_t proposal_id;
    common::LogIndex log_index;
    common::Status status;
    bool final_result;  // always true in T023
};
```

- Buffered in `std::vector<ProposalResult>` protected by `std::mutex`.
- Consumed by `CollectProposalResults()` (thread-safe, clears buffer).
- Non-leader proposals → immediate failure result.
- Leader proposals → result only after commit + apply success.
- Apply failure → failure result, `appliedIndex` does not advance.
- Result queue full → buffer is unbounded (vector) — no silent drops.

## Proposal Success/Failure Semantics

| Scenario | Result |
|----------|--------|
| Non-leader proposal | Immediate failure |
| Leader proposal, log append fails | Immediate failure |
| Entry appended but not committed | No result (pending) |
| Entry committed but not applied | No result (pending) |
| Entry committed and applied successfully | Success result |
| Entry committed but apply failed | Failure result |

## RaftCore Changes

Two additions:

1. **`ConfirmApplied(LogIndex index)`**: Validates sequential order and advances `appliedIndex` via `RaftLog::AdvanceAppliedTo()`.
2. **`TryAdvanceCommitIndex()` in `Propose()`**: After a leader appends a proposal, the system checks if the entry can be committed. This enables single-node clusters to commit immediately.

No existing interfaces or behaviors were modified. No member variables were added.

## Deviations, Blockers, or Unfinished

- No blockers.
- `TryAdvanceCommitIndex()` was added to `Propose()` to enable single-node commit. Without this, a single-node leader could never commit its own proposals, breaking the apply pipeline. This is a minimal and correct change — the same logic already existed in `HandleAppendEntriesResponse`.
