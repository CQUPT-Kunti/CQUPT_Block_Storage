## Peer Queue Creation and Capacity

During `Initialize()`:
1. Reads `core_->voter_ids()` and `core_->learner_ids()`.
2. Collects all unique remote peer IDs (excluding `core_->node_id()`).
3. Creates a `PeerMessageQueue` (deque + mutex + closed flag) for each peer with capacity from `Options::peer_queue_capacity`.
4. Capacity must be > 0; zero returns `InvalidArgument`.

## Message Distribution by Peer

`DistributeOutputMessages()` is called from `DrainCoreOutput()`:
- Each output message has a `target_node_id`.
- Messages addressed to self are skipped.
- Messages are pushed to the specific peer's queue.
- Unknown peers return `NotFound`.
- Full queue returns `ResourceExhausted`.
- Both immediate and persisted messages are distributed after their respective gates.

## Per-Peer Backpressure Isolation

- Each peer has its own `PeerMessageQueue` with independent capacity.
- When one peer's queue is full, only that peer's `DistributeOutputMessages` returns `ResourceExhausted`.
- Other peers remain unaffected.
- No shared global message buffer is used.

## Runtime Lifecycle States

```
CREATED → INITIALIZED → RUNNING → STOPPING → STOPPED
```

| State | Meaning |
|-------|---------|
| `CREATED` | Constructed. No initialization performed. |
| `INITIALIZED` | `Initialize()` called. Peer queues set up. Threads not started. |
| `RUNNING` | `Start()` called. Three worker threads running. Events accepted. |
| `STOPPING` | `RequestShutdown()` called. Queues closed. Threads exiting. |
| `STOPPED` | `WaitForShutdown()` returned. Threads joined. |

## Shutdown Ordering

1. `RequestShutdown()` sets state to `STOPPING`.
2. Closes `event_queue_` → protocol loop exits.
3. Closes `persistence_queue_` → persistence loop exits.
4. Closes `apply_queue_` → apply loop exits.
5. Closes all peer queues via `CloseAllPeerQueues()`.
6. `WaitForShutdown()` joins protocol, persistence, and apply threads.
7. State advances to `STOPPED`.

## Queue Drain/Rejection Rules

| Queue | Behavior on Shutdown |
|-------|---------------------|
| Event queue | Closed. New events return `Busy`. In-flight events complete. |
| Persistence queue | Closed. Started work completes. Unstarted work is not executed. |
| Apply queue | Closed. Started apply completes. Unapplied entries produce no result. |
| Peer queues | Closed. Remaining messages can still be dequeued. New pushes return `Busy`. |
| Proposal results | Bounded. Full drops are silent (documented limitation). |

## Thread Join and Destructor Safety

- `WaitForShutdown()` checks `joinable()` on each thread before joining.
- Destructor checks if state is `RUNNING` or `STOPPING` before calling shutdown.
- No `detach()` is used.
- No sleep or real-time waits are used for synchronization.
- Repeated `WaitForShutdown()` is safe.

## ProposalResult Bounded Buffer

T023 used an unbounded `std::vector<ProposalResult>`. T024 replaces this with `ResultQueue`:

```cpp
struct ResultQueue {
    std::deque<ProposalResult> items;
    std::size_t max_capacity;
};
```

- Capacity configured via `Options::proposal_result_queue_capacity`.
- When full, new results are silently dropped (documented limitation; proper backpressure requires a consumer that drains the queue regularly).
- `CollectProposalResults(max_count)` non-blockingly drains up to `max_count` items.

## Deviations, Blockers, or Unfinished

- No blockers.
- `CollectOutputMessages()` was removed in favor of `TryDequeuePeerMessage()` per peer. Messages are no longer collected in a single flat buffer but are routed directly to peer queues during `DrainCoreOutput()`.
- Peer queues use a custom `PeerMessageQueue` (deque + mutex) instead of `BoundedQueue` because `BoundedQueue::Pop()` blocks on empty, which is unsuitable for the non-blocking `TryDequeuePeerMessage()` API.
- Similarly, `ProposalResult` uses a custom `ResultQueue` (deque + mutex) instead of `BoundedQueue` for the same non-blocking Pop requirement.
- Peer messages are only generated by RaftCore's `StageMessage()` path (e.g., RequestVote/AppendEntries responses). Leader heartbeat/append replication requires explicit `BuildAppendEntriesForPeer()` calls, which will be added in T028/T029 (Transport layer).
- ProposalResult drops are silent when the bounded buffer is full. In production, the consumer (`CollectProposalResults`) should be called frequently enough to prevent drops.
