## Created Proto Files

### `common.proto`

Shared data structures reused across protocol files:

- `RpcStatusCode` and `RpcStatus` for client/control-plane result shells
- `NodeAddress` and `LeaderHint` for endpoint and redirect information
- `ClusterRole` and `NodeLifecycleState` for status reporting
- `ClusterMember`, `MembershipView`, `HardState`, `PeerProgress`
- `LogEntry` and `SnapshotMetadata` so `raft.proto` can reuse existing Raft payload semantics directly

### `raft.proto`

Peer-to-peer Raft contract only:

- `RaftService` with `RequestVote`, `AppendEntries`, and `InstallSnapshot`
- Request/response messages matching `RequestVoteRequest/Response`, `AppendEntriesRequest/Response`, and `InstallSnapshotRequest/Response`
- `VoteRejectReason` aligned with current `raft_message.h`

### `metadata.proto`

Client-facing Metadata service shell:

- `MetadataService` with `Propose`, `Query`, `GetLeader`, `GetStatus`, `AddLearner`, `PromoteLearner`, and `RemoveMember`
- `Propose` and `Query` keep payloads opaque as `bytes`
- `GetStatus` returns a compact `RaftNodeStatus` shell based on current Raft-visible state
- Membership management requests use node/member identifiers only, without embedding consensus logic

### `store.proto`

Store control-plane shell:

- `StoreControlService` with `Register`, `Heartbeat`, `Stop`, `Remove`, `PollTasks`, and `ReportTaskResult`
- `StoreDescriptor` for registration/state transitions
- `TaskAssignment` and task enums as protocol shells only
- Heartbeat remains leader-local and does not define durable business behavior

## Mapping to Existing Interfaces

- `raft.proto` maps to current in-memory Raft message payload structs one-to-one at the semantic level.
- `metadata.proto` maps to current phase boundaries only:
  - `Propose` corresponds to `RaftRuntime::EnqueueProposal` semantics
  - `GetLeader` / `GetStatus` correspond to externally visible node and leader state already exposed by `RaftCore`/`RaftRuntime`
  - membership RPCs are shells only and do not implement T031+ behavior
- `store.proto` maps to specification-level Metadata example contracts only:
  - durable transitions use request/response shells
  - heartbeats remain transient shells
  - task polling/result reporting keep payloads opaque and idempotency-oriented

## Scope and Boundary Decisions

- No gRPC service class, RPC adapter, transport, connection management, or generated business code was implemented.
- No Raft algorithm logic was moved into proto.
- No `RuntimeEvent`, `PersistenceWork`, `ApplyWork`, storage internals, or thread model details were exposed.
- No business command/query payload format was defined; those remain opaque `bytes`.

## CMake Integration

Minimal proto-only integration was added:

- `cross-platform-raft/proto/CMakeLists.txt` defines `cpr_proto_contracts` as a source-only custom target
- `cross-platform-raft/CMakeLists.txt` adds `add_subdirectory(proto)`

This keeps the `.proto` files attached to the build graph without:

- downloading protobuf or gRPC
- generating `.pb.cc` / `.grpc.pb.cc`
- linking generated code into runtime, RPC, server, or tests
