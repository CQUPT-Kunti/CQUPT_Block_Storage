## Transport Interface Design

`IRaftTransport` exposes only four operations:

- `Initialize(local_node_id, peers)`
- `Start()`
- `Stop()`
- `Send(message, response)`

Design points:

- Public interface exposes only project types: `Status`, `NodeId`, `RaftMember`, `RaftMessage`
- No gRPC types are exposed
- No channel, socket, fd, or thread handle is exposed
- `Send()` is explicitly unary request/response oriented
- Response-type `RaftMessage` values are rejected as invalid outbound transport inputs

`RaftTransportOptions` currently carries only one setting:

- `rpc_timeout_ms`

## GrpcRaftTransport Implementation

`GrpcRaftTransport` is a synchronous implementation over T027 generated gRPC code:

- Internal peer map: `NodeId -> RaftMember`
- Internal stub map: `NodeId -> cpr::raft::v1::RaftService::Stub`
- Internal state machine:
  - `CREATED`
  - `INITIALIZED`
  - `STARTED`
  - `STOPPED`

Behavior:

- `Initialize()` validates local id, peer ids, host, port, and duplicate peers
- Self peer entries are ignored
- `Start()` is idempotent
- `Stop()` is idempotent
- Destructor calls `Stop()` safely
- No background thread is created
- No retry policy is implemented

## Lifecycle Design

- `Send()` before `Start()` returns `Busy`
- `Send()` after `Stop()` returns `Busy`
- Unknown target peer returns `NotFound`
- Response-type outbound messages return `InvalidArgument`
- Remote unary RPC failures are converted into project `Status`
  - `UNAVAILABLE` / `DEADLINE_EXCEEDED` → `RetryLater`
  - `INVALID_ARGUMENT` → `InvalidArgument`
  - `NOT_FOUND` → `NotFound`
  - `RESOURCE_EXHAUSTED` → `ResourceExhausted`
  - other gRPC failures → `InternalError`
