## RaftRPCServiceAdapter Design

`RaftRpcServiceAdapter` derives directly from T027 generated `cpr::raft::v1::RaftService::Service`.

Public surface:

- constructor with `RaftRuntime *`
- constructor with `RaftRuntime *` + small `Options`
- overrides for `RequestVote`
- overrides for `AppendEntries`
- overrides for `InstallSnapshot`

`Options` contains only:

- `response_timeout`
- `poll_interval`

No extra public interface, no callback/reactor wrapper, no listener address, no server ownership, no background thread.

## Implemented RPC Methods

All `raft.proto` node RPC methods are covered:

- `RequestVote`
- `AppendEntries`
- `InstallSnapshot`

No `MetadataService` or `StoreService` implementation was added.

## Protobuf to Internal Message Conversion

Conversion is local to `raft_rpc_service.cpp`.

Mappings implemented:

- `RequestVoteRequest` -> `raft::RequestVoteRequest`
- `AppendEntriesRequest` -> `raft::AppendEntriesRequest`
- `InstallSnapshotRequest` -> `raft::InstallSnapshotRequest`
- repeated proto `LogEntry` -> `std::vector<raft::LogEntry>`
- proto `SnapshotMetadata` / `MembershipView` -> internal `SnapshotMetadata` / `MembershipView`

Validation kept minimal and explicit at the RPC boundary:

- invalid `candidate_id`
- invalid `leader_id`
- invalid `prev_log_index` + `prev_log_term` combination
- invalid snapshot index/term

No Protobuf type was introduced into `RaftCore`, `RaftRuntime`, `RaftLog`, or storage public interfaces.

## Runtime Delivery

Inbound RPC handling path:

1. Validate protobuf request
2. Convert to `raft::RaftMessage`
3. Call `RaftRuntime::EnqueueMessage(source_node_id, message)`
4. Poll `RaftRuntime::TryDequeuePeerMessage(source_node_id, ...)`
5. Convert the returned internal response back to protobuf

This keeps all request execution inside the existing runtime thread model. The adapter does not call `RaftCore` directly.

## Business Error vs Transport Error Mapping

Business-level Raft rejections stay inside normal OK RPC responses:

- stale term vote rejection
- append rejection
- snapshot rejection with `success = false`

Non-OK gRPC status is returned only for adapter/runtime failures such as:

- null runtime dependency -> `UNAVAILABLE`
- runtime not running -> `FAILED_PRECONDITION`
- enqueue failure -> mapped from project `Status`
- peer response timeout -> `UNAVAILABLE`
- unexpected response type -> `INTERNAL`

Project `Status` to gRPC mapping is intentionally small and consistent:

- `InvalidArgument` -> `INVALID_ARGUMENT`
- `NotFound` -> `NOT_FOUND`
- `Busy` -> `FAILED_PRECONDITION`
- `ResourceExhausted` -> `RESOURCE_EXHAUSTED`
- `RetryLater` -> `UNAVAILABLE`
- `IoError` / `Corruption` -> `DATA_LOSS`
- `InternalError` -> `INTERNAL`

## Lifecycle and Thread Boundary

- The adapter does not start or stop `gRPC::Server`
- The adapter does not listen on any port
- The adapter does not create client channels
- The adapter does not create background threads
- The adapter only waits synchronously for a response already produced by runtime processing

## Minimal Runtime Change

`RaftRuntime` originally had no inbound `InstallSnapshot` handling path.

Minimal change added in `raft_runtime.cpp`:

- handle `RaftMessageType::INSTALL_SNAPSHOT_REQUEST`
- persist snapshot through existing storage boundary
- emit an `INSTALL_SNAPSHOT_RESPONSE` through the existing peer queue path
- ignore `INSTALL_SNAPSHOT_RESPONSE` on inbound runtime processing

This change was required because T029 needed to cover all RPC methods in `raft.proto`, while the existing runtime only handled vote and append messages.
