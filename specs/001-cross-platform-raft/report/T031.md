## Membership State Model

`MembershipState` is a small value-style state holder with:

- sorted `voters`
- sorted `learners`
- `has_active_transition`
- `configuration_id`

The state exposes only the queries T031 needs:

- `IsVoter()`
- `IsLearner()`
- `Contains()`
- `CanVote()`
- `CountsTowardQuorum()`
- `CanBecomeLeader()`
- `ToView()`
- `ToLogEntry()`
- `ApplyCommitted()`

This keeps learner restrictions expressible without modifying `RaftCore` election or quorum logic in T031.

## Voters and Learners Representation

- Representation uses `std::vector<RaftMember>`.
- Members are normalized into deterministic order by `node_id`, then host, then port.
- Validation rejects:
  - `node_id == 0`
  - duplicate voter IDs
  - duplicate learner IDs
  - overlap between voter and learner sets
  - empty voter sets
  - `configuration_id == 0`

Empty voter sets were explicitly disallowed for bootstrap, restore, and committed-configuration apply paths. This behavior is covered by tests.

## Learner Eligibility Constraints

Learner restrictions are represented by query semantics only:

- learners return `false` from `CanVote()`
- learners return `false` from `CountsTowardQuorum()`
- learners return `false` from `CanBecomeLeader()`

No T032-T035 workflow, quorum rewrite, or election behavior change was added.

## Active Transition State

- Transition state is represented by one boolean: `has_active_transition`.
- This keeps the T031 model able to distinguish stable vs transition configuration.
- Because the model stores only one current committed configuration at a time, the “at most one active transition” rule is naturally enforced.

The module does not implement transition planning, add/promote/remove workflows, or two-phase execution.

## Membership Log Entry Design

`MembershipLogEntry` is an internal raft-only payload model with:

- `configuration_id`
- `voters`
- `learners`
- `has_active_transition`

This is paired with existing `LogEntryType::MEMBERSHIP_CHANGE` to distinguish membership entries from normal command entries.

The design stores the committed membership view itself instead of an operation script. That is enough for T031 apply/idempotence/recovery semantics and leaves T032-T035 free to decide how proposals are produced later.

## Serialization Format and Versioning

`EncodeMembershipLogEntry()` and `DecodeMembershipLogEntry()` use a deterministic binary format:

1. `u8` format version
2. `u8` flags
3. `u64` configuration id
4. encoded voter list
5. encoded learner list

Each member encodes:

- `u64 node_id`
- `u32 host_length`
- raw host bytes
- `u16 port`

Current rules:

- version `1` only
- flag bit `0x01` means active transition
- unknown version -> `CORRUPTION`
- unknown flags -> `CORRUPTION`
- truncated bytes -> `CORRUPTION`
- invalid decoded membership content -> `CORRUPTION`

The format lives entirely inside raft and does not introduce protobuf, JSON, or metadata business types.

## Configuration ID and Idempotence

- `configuration_id` is required to be non-zero.
- `ApplyCommitted()` rejects rollback when the incoming id is less than the current id.
- Reapplying the same `configuration_id` with identical normalized content is treated as idempotent success.
- Reapplying the same `configuration_id` with different content is rejected.

This gives T031 a monotonic committed-membership identity without changing `HardState`.

## Bootstrap Rules

`MembershipState::Bootstrap()` requires:

- valid local node id
- valid bootstrap membership view
- local node must be part of the initial configuration
- `HardState.membership_configuration_id == 0`
- no snapshot present

If hard-state membership or snapshot membership already exists, bootstrap returns `BUSY` and does not overwrite runtime truth.

## Persistent View and Recovery

Persistence reuse is direct:

- `MembershipState::ToView()` produces `MembershipView`
- `MembershipState::FromView()` restores from `MembershipView`
- `SnapshotMetadata.membership` remains the snapshot carrier
- `HardState.membership_configuration_id` remains the comparable committed id

No storage, snapshot, or runtime persistence flow was reimplemented.

## Atomic Update Strategy

State updates use validate-then-commit:

- input view or log entry is validated first
- normalized candidate state is built in a temporary object
- only then is the visible state replaced

On failure, the original `MembershipState` remains unchanged.
