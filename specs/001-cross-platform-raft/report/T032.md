## AddLearner Internal Entry Design

Added a direct internal entry point on `RaftCore`:

- `AddLearner(const RaftMember&, LogIndex*)`

Behavior:

- only `LEADER` may call it
- `FOLLOWER`, `CANDIDATE`, and `LEARNER` return explicit `InvalidArgument`
- proposal appends a normal Raft log entry with:
  - `type = LogEntryType::MEMBERSHIP_CHANGE`
  - current leader term
  - encoded `MembershipLogEntry` payload
- committed membership is **not** modified at proposal time
- `PeerProgress` for the new learner is **not** created at proposal time

This keeps AddLearner log-driven instead of mutating runtime membership locally.

## AddLearner Validation Rules

Validation lives primarily in `BuildAddLearnerLogEntry()` inside `raft_membership.cpp`.

Rejected cases:

- invalid `node_id`
- empty host
- zero port
- already-present voter
- already-present learner
- address conflict with any existing member
- current committed membership has `has_active_transition = true`
- empty/unavailable membership state
- any resulting invalid membership view after appending the learner

On failure:

- `MembershipState` is unchanged
- `RaftLog` is unchanged
- `PeerProgress` is unchanged

## Membership Log Generation

`raft_membership` now provides:

- `BuildAddLearnerLogEntry(const MembershipState&, const RaftMember&, MembershipLogEntry*)`

Generation rules:

- preserves existing voters
- preserves existing learners
- appends the new node only to learners
- never inserts directly into voters
- increments `configuration_id` by `+1`
- keeps `has_active_transition = false` for T032
- normalizes member ordering deterministically before encoding

The encoded payload still uses the T031 binary format and versioning scheme.

## Configuration ID Progression

- committed configuration id still comes from `MembershipState`
- AddLearner proposal computes `new_id = current_id + 1`
- duplicate apply of the same committed membership entry remains idempotent through T031 `ApplyCommitted()`
- rollback to an older configuration id is still rejected

`HardState.membership_configuration_id` is updated only when a committed membership entry is applied or when snapshot membership metadata is accepted, not when a proposal is merely appended.

## Membership Log Commit/Application Path

Added `RaftCore::ApplyCommittedMembershipEntry(const LogEntry&)`.

Behavior:

1. Require `LogEntryType::MEMBERSHIP_CHANGE`
2. Decode `MembershipLogEntry`
3. Apply it into a temporary `MembershipState`
4. Build temporary voter/learner id sets
5. Build temporary `PeerProgress` map
6. Commit all updates only after validation succeeds

This keeps committed membership changes atomic and failure-safe.

## Learner PeerProgress Initialization and Update

When a committed AddLearner entry is applied:

- a new learner `PeerProgress` is created if absent
- `is_learner = true`
- `match_index = 0`
- `next_index = last_index + 1`
- `is_active = false` initially

Update behavior reuses existing replication logic:

- `HandleAppendEntriesResponse()` still updates learner `match_index` and `next_index` the same way as follower progress
- learner `is_active` becomes `true` after a successful or rejected response is processed
- `HandleInstallSnapshotResponse()` now updates learner progress after a snapshot catch-up step:
  - `match_index = max(old, last_included_index)`
  - `next_index = match_index + 1`

No separate learner-only replication algorithm was introduced.

## Log or Snapshot Catch-Up Decision

Added:

- `GetReplicationModeForPeer(node_id, ReplicationMode*)`
- `BuildInstallSnapshotForPeer(node_id, InstallSnapshotRequest*)`
- existing `BuildAppendEntriesForPeer()` remains the append path

Decision rule:

- if `peer.next_index <= log_.snapshot_index()`: choose `INSTALL_SNAPSHOT`
- otherwise: choose `APPEND_ENTRIES`

Snapshot build behavior:

- uses current term
- uses current leader id
- uses `log_.snapshot_index()` and `log_.snapshot_term()`
- includes current committed membership view in snapshot metadata
- leaves snapshot payload empty in T032, because T032 only expresses RaftCore catch-up intent and does not reimplement snapshot storage or transport

After `HandleInstallSnapshotResponse()` succeeds, the learner can continue through `BuildAppendEntriesForPeer()` from the snapshot boundary forward.

## Learner Health and Caught-Up State

Added:

- `GetLearnerState(node_id, LearnerState*)`

Reported fields:

- `healthy`
- `caught_up`
- `requires_snapshot`
- `match_index`
- `next_index`
- `leader_commit`

Rules:

- `healthy` uses existing `PeerProgress.is_active`
- `caught_up` is true only when:
  - learner does not require snapshot
  - learner `match_index >= leader commit_index`
- no wall-clock or monotonic time is read
- no new timer, health monitor, or runtime thread was added

## Learner Non-Quorum Guarantee

The quorum guarantee remains in existing commit logic:

- `TryAdvanceCommitIndex()` still uses only `voter_ids_`
- learner `PeerProgress` is updated, but learner entries are never counted toward majority
- learner still cannot start an election
- learner still cannot become leader
- learner still cannot vote

T032 did not relax any of those constraints.

## Atomic Update and Rollback Strategy

Atomicity is handled by temporary state construction:

- `MembershipState` candidate first
- voter/learner id vectors next
- `PeerProgress` map candidate next
- visible `RaftCore` state replaced only after all validation succeeds

Failure leaves:

- committed membership unchanged
- peer progress unchanged
- configuration id unchanged
