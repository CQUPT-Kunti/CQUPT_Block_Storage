## PromoteLearner Internal Entry

Added a direct internal entry point on `RaftCore`:

- `PromoteLearner(NodeId learner_node_id, LogIndex* log_index = nullptr)`

Behavior:

- only `LEADER` may call it
- rejects invalid node id
- rejects while another membership change is still pending
- rejects when the committed voter configuration no longer has quorum
- rejects when the target is not a committed learner
- rejects when the learner is unhealthy, still requires snapshot catch-up, or has not caught up to the leader `commit_index`
- proposal appends a normal `LogEntryType::MEMBERSHIP_CHANGE` entry instead of mutating committed membership directly

## RemoveMember Internal Entry

Added:

- `RemoveMember(NodeId member_node_id, LogIndex* log_index = nullptr)`

Behavior:

- only `LEADER` may call it
- rejects invalid node id
- rejects while another membership change is pending
- rejects when the committed voter configuration has lost quorum
- rejects unknown nodes
- rejects removing the last voter
- supports removing either a committed learner or a committed voter
- removal takes effect only after the membership log entry is committed and applied

## Membership Validation and Log Generation

`raft_membership` now provides:

- `BuildPromoteLearnerLogEntry(...)`
- `BuildRemoveMemberLogEntry(...)`

Rules:

- both operate on committed `MembershipState`
- both reject `has_active_transition = true`
- promote moves exactly one node from `learners` to `voters`
- remove deletes exactly one node from either `voters` or `learners`
- `configuration_id` always advances by `+1`
- entry content is normalized through the existing deterministic T031 logic
- no new persistence file, side state, or alternate membership encoding was introduced

## One-Change-at-a-Time Enforcement

Added:

- `HasPendingMembershipChange()`
- shared helper `AppendMembershipChangeProposal(...)`

Enforcement:

- `AddLearner`, `PromoteLearner`, and `RemoveMember` all reject with `Busy` when a previous membership-change log entry is still pending
- failed requests do not occupy the slot because no membership log entry is appended on validation failure
- proposal generation is centralized in one helper so all three paths use the same append and encoding behavior

Current boundary:

- pending detection is log-based and clears after the membership entry is applied and `ConfirmApplied()` advances the applied boundary
- T033 does not add a separate uncommitted-membership tracker or request-correlation framework

## Old-Config Quorum Gate

Added:

- `HasCommittedVoterQuorum()`

Rules:

- quorum is computed only from committed `voter_ids_`
- the local voter counts as active
- remote voters count only when existing `PeerProgress.is_active` is true
- learners never count toward quorum
- `PromoteLearner` and `RemoveMember` reject when the old committed voter set has already lost quorum

This preserves the T033 requirement that learner catch-up does not bypass old-configuration safety.

## Membership Commit and Apply Behavior

T032 `ApplyCommittedMembershipEntry(...)` remains the apply path. T033 extends its observable behavior by relying on that atomic apply path for both promote and remove:

- decode the committed `MembershipLogEntry`
- validate against current committed membership
- build temporary membership vectors and `PeerProgress` map
- replace visible state only after validation succeeds
- keep repeated application of the same `configuration_id` idempotent
- reject conflicting content for an already-applied `configuration_id`

No promote or remove request mutates committed membership during proposal time.

## Local Leader Removal and Local Role Safety

When a committed membership update removes the local node from voters, or leaves it only as a learner:

- local `role_` is forced to `LEARNER`
- `leader_id_` resets to invalid
- `hard_state_.voted_for` resets to invalid
- election ticks are reset

The same demotion guard is also applied after snapshot-boundary membership refresh so the local node does not continue acting as a voter or leader after removal.

## Learner Non-Quorum Guarantee

T033 keeps the T032 learner constraints unchanged:

- learners still receive and track replication progress
- learners still do not vote
- learners still do not count toward majority commit advancement
- learners still cannot become leader

No quorum path was changed to include learners.

## Atomicity and Failure Rollback

Atomicity still relies on temporary-state construction before commit:

- candidate `MembershipState`
- candidate voter/learner vectors
- candidate `PeerProgress` map

If validation fails:

- committed membership remains unchanged
- `PeerProgress` remains unchanged
- `configuration_id` remains unchanged
- no membership log entry is appended for rejected requests
