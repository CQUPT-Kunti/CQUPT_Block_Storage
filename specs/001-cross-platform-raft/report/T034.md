## Safe Transition Design

T034 uses a persisted two-phase membership transition for voter-set changes:

1. **Phase 1 transition entry**
   - keeps the current committed voter set in `voters`
   - keeps the current learner set in `learners`
   - sets `has_active_transition = true`
   - persists the target stable configuration in `next_voters` and `next_learners`
2. **Phase 2 final entry**
   - clears `has_active_transition`
   - moves `voters` and `learners` to the target stable configuration
   - clears `next_voters` and `next_learners`

`AddLearner` still uses a single committed entry because it does not change the voter quorum. `PromoteLearner` and `RemoveMember(voter)` now go through transition then finalization. `RemoveMember(learner)` stays single-stage.

## Old Config, New Config, and Active Transition State

Internal `MembershipView` and `MembershipLogEntry` were minimally extended with:

- `next_voters`
- `next_learners`

Meaning:

- stable config: `has_active_transition = false`, `next_*` empty
- active transition: `voters`/`learners` describe the currently committed pre-final state, `next_*` describe the committed target stable state

This keeps transition semantics recoverable without changing protobuf contracts.

## Phase 1 and Final Log Generation

`raft_membership.cpp` now provides:

- `BuildAddLearnerLogEntry(...)`
- `BuildPromoteLearnerLogEntry(...)`
- `BuildRemoveMemberLogEntry(...)`
- `BuildFinalizeMembershipLogEntry(...)`

Rules:

- `PromoteLearner` phase 1 writes current config plus target voter/learner sets
- `RemoveMember(voter)` phase 1 writes current config plus target voter set without the removed voter
- `FinalizeMembershipChange()` generates the stable second-stage entry from the committed active transition
- `configuration_id` stays monotonic: phase 1 uses `current + 1`, phase 2 uses `phase1 + 1`
- serialization is deterministic and versioned

## Active Transition Quorum Rule

During `has_active_transition = true`, `RaftCore::TryAdvanceCommitIndex()` no longer uses only the old median-of-voters rule.

It now:

- checks the entry term is current
- requires majority of the old voter set
- requires majority of `next_voters`

This is the T034 safe transition point:

- learners still do not count before phase 1
- promoted learner only starts participating after phase 1 is committed and applied
- removed voter still participates through the old side until final config applies

## Leader Change and Recovery

Committed transition state survives in two places:

- membership log payloads
- snapshot/persistent membership view through internal `MembershipView.next_*`

Recovery behavior:

- `MembershipState::FromView()` restores both stable and active-transition state
- `DecodeMembershipLogEntry()` recognizes old version 1 entries and new version 2 transition entries
- `ApplyCommitted()` preserves active transition state and rejects conflicting same-`configuration_id` content

This means a restarted node or a new leader does not forget that a committed transition is in progress.

Current boundary:

- no automatic “new leader resumes finalization” workflow was added
- the committed active transition is preserved, and a leader can safely call the existing finalize path later

## Membership Slot Release Rule

T033’s one-change-at-a-time rule was extended:

- phase 1 proposal pending in log: slot occupied
- phase 1 applied and active transition present: slot occupied
- phase 2 proposed but not yet applied: slot occupied
- phase 2 applied and stable config reached: slot released

Failed validation does not consume the slot.

## Runtime Membership Event and Queue

`RaftRuntime` now implements `EnqueueMembershipChange(...)` for real.

Design:

- RPC thread enqueues `MembershipChangeEvent`
- protocol thread executes the corresponding `RaftCore` call
- runtime tracks the log index of the accepted membership request
- on `ApplyCompletionEvent`, membership entries are applied through `RaftCore::ApplyCommittedMembershipEntry(...)`
- if phase 1 belongs to `PromoteLearner` or voter removal, runtime issues `FinalizeMembershipChange()` on the protocol thread

No extra protocol thread, no detached worker, no generic workflow state machine.

## Membership Result Correlation and Completion Semantics

Added minimal result tracking:

- `MembershipChangeEvent.request_id`
- `MembershipChangeResult`
- `RaftRuntime::TryTakeMembershipChangeResult(...)`

Semantics:

- enqueue success does not mean membership success
- `AddLearner` result is emitted only after the committed membership entry is applied
- `PromoteLearner` / voter removal result is emitted only after phase 2 stable config is applied
- failure during apply or finalization emits a final failure result
- runtime stop prevents further RPC enqueue and waiting succeeds only while runtime is alive

This is request/result correlation, not a reusable workflow framework.

## Metadata RPC Adapter Design

Created `MetadataRpcServiceAdapter` deriving directly from generated `cpr::metadata::v1::MetadataService::Service`.

Implemented only:

- `AddLearner`
- `PromoteLearner`
- `RemoveMember`

Other Metadata RPCs remain the generated default unimplemented behavior because T034 scope is membership only.

Responsibilities kept in RPC:

- parameter validation
- protobuf -> `MembershipChangeEvent` conversion
- runtime enqueue
- bounded wait on `request_id`
- result -> protobuf response mapping

RPC does not compute quorum, catch-up status, or membership logs.

## Non-Leader Response and Leader Address

When the runtime protocol thread sees a membership request on a non-leader:

- result status is emitted as business rejection
- `MetadataRpcServiceAdapter` returns gRPC `OK`
- response `status.code` becomes `RPC_STATUS_CODE_NOT_LEADER`
- leader hint includes `leader_id`
- leader address is resolved from the committed membership view when known

Unknown leader stays explicit by leaving the hint unset or address empty.

## Business Error vs gRPC Error Mapping

Business rejections returned in normal protobuf response:

- `NOT_LEADER`
- `BUSY`
- `NOT_FOUND`
- `FAILED_PRECONDITION`-style safety or state rejection through response status code/message

Non-OK gRPC status is reserved for transport/runtime boundary failures:

- invalid RPC arguments
- runtime not running
- queue full
- deadline exceeded
- client cancellation
- internal/data-loss style failures

## Persistence Gating and Final Success Condition

Success is not reported at proposal time.

The runtime only completes a membership request after:

1. the membership log entry is persisted and committed by existing Raft flow
2. apply completion arrives in order
3. `RaftCore::ApplyCommittedMembershipEntry(...)` succeeds
4. for two-phase changes, phase 2 stable config is also committed and applied

This preserves the T034 requirement that success is not returned from memory-only or local-persist-only state.

## Atomic Membership and PeerProgress Update

`RaftCore::ApplyCommittedMembershipEntry(...)` still uses validate-then-commit:

- decode and validate the entry
- build candidate `MembershipState`
- build candidate voter/learner vectors
- build candidate `PeerProgress`
- replace visible state only after all checks succeed

The only extension is that active-transition application now uses union voter ids for runtime/replication tracking while preserving `next_*` target semantics in committed membership state.
