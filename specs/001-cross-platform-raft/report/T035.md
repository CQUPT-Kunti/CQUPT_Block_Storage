## Test Fixture and Message Driving

`raft_membership_test.cpp` adds a small file-local fixture style based on helper functions only:

- `MakeMembership(...)`, `MakeOptions(...)`, `MakeCore(...)`
- `AcceptReplication(...)`, `RejectReplication(...)`
- `ApplyCommittedMembershipAt(...)`
- `CommitLearnerAdd(...)`
- `CommitCommandAndCatchUpLearner(...)`

The tests drive membership state only through public `RaftCore` and `MembershipState` interfaces. Replication progress is advanced deterministically by explicit `AppendEntriesResponse` and `InstallSnapshotResponse` injection. No wall-clock logic, fixed port, external service, or hidden hook is used.

## Simulation Path Design

The consolidated file uses the direct simulation path for deterministic stage validation:

- commit `AddLearner` through membership log apply
- advance learner catch-up through append or snapshot response
- execute `PromoteLearner` phase 1 and finalization phase 2
- execute `RemoveMember` phase 1 and phase 2
- rebuild a core from persisted transition `MembershipView` to validate recovery semantics

This keeps the tests close to the T031-T034 public consensus boundary and avoids inventing a generic cluster harness.

## Learner Eligibility Tests

Covered in `RaftMembershipTest.LearnerLifecycleUsesLogThenTwoPhasePromotion`:

- learner cannot vote before transition
- learner does not count toward quorum before transition
- learner cannot become leader before transition
- learner enters only `learners` after committed `AddLearner`
- learner becomes quorum-eligible only after the committed transition phase applies

Covered in existing retained suite:

- learner election/voting/quorum query semantics in `raft_membership_state_test.cpp`

## Learner Log and Snapshot Catch-Up Tests

Covered in `RaftMembershipTest.SnapshotCatchUpBlocksPromotionUntilLearnerIsHealthyAndCaughtUp`:

- rejected append drives replication mode to `INSTALL_SNAPSHOT`
- `BuildInstallSnapshotForPeer()` carries snapshot boundary and membership metadata
- successful snapshot response switches the learner back to append catch-up
- learner promotion remains rejected until healthy and caught up

Covered in `RaftMembershipTest.LearnerLifecycleUsesLogThenTwoPhasePromotion`:

- append-driven catch-up path
- learner reaches caught-up state before promotion

## PromoteLearner Two-Phase Tests

Covered in `RaftMembershipTest.LearnerLifecycleUsesLogThenTwoPhasePromotion`:

- phase 1 applies an active transition configuration
- phase 1 keeps the node in `learners` while `next_voters` includes it
- phase 2 finalization clears active transition
- final stable config moves the node into `voters`
- `configuration_id` advances monotonically from 1 -> 2 -> 3 -> 4 in the full add/promote flow

## RemoveMember Tests

Covered in `RaftMembershipTest.QuorumLossRejectsUnsafePromoteAndRemoveWhileAllowingManualReplacementWithQuorum`:

- manual failed-voter replacement uses promote phase 1 + phase 2, then remove-voter phase 1 + phase 2
- final stable config contains replacement voter and excludes the failed voter
- removed voter `PeerProgress` is removed

Covered in `RaftMembershipTest.RemoveCurrentLeaderDemotesLocallyOnlyAfterFinalStableConfig`:

- removing the current leader does not demote locally during phase 1
- local node is demoted only after final stable configuration applies

## Single Membership Change Tests

Covered in `RaftMembershipTest.SingleMembershipChangeSlotBlocksConcurrentRequestsAndRecoveryKeepsBusyState`:

- `AddLearner` proposal blocks concurrent `RemoveMember` and `PromoteLearner`
- committed active transition still blocks a second membership request
- reconstructed core from persisted transition state still rejects a second membership request
- the slot is released only after final stable configuration applies

## Permanently Failed Voter Replacement Scenario

Covered in `RaftMembershipTest.QuorumLossRejectsUnsafePromoteAndRemoveWhileAllowingManualReplacementWithQuorum`:

1. start from a 3-voter configuration
2. manually add learner 4
3. catch it up
4. promote it through two safe phases
5. remove failed voter 3 through two safe phases
6. verify no automatic replacement or bypass of Raft log was used

This is the T035 permanent-failure replacement path with quorum still available.

## Quorum-Loss Rejection Scenario

Covered in `RaftMembershipTest.QuorumLossRejectsUnsafePromoteAndRemoveWhileAllowingManualReplacementWithQuorum`:

- when the old voter configuration has lost quorum, `PromoteLearner` is rejected
- when the old voter configuration has lost quorum, `RemoveMember(voter)` is rejected
- learner membership is not enough to fake old-config quorum
