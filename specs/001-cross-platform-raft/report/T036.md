## IRaftStateMachine Interface Responsibilities and Signature

`IRaftStateMachine` is defined in `state_machine.h` as a pure abstract boundary with exactly three responsibilities:

- `Apply(LogIndex index, Term term, const OpaquePayload &command_payload)`
- `CreateSnapshot(LogIndex last_applied_index, Term last_applied_term, OpaquePayload *snapshot_payload)`
- `RestoreSnapshot(const OpaquePayload &snapshot_payload)`

No other behavior was added:

- no `Propose`
- no `Query`
- no `GetLeader`
- no `GetStatus`
- no store/placement/task API
- no queue/thread lifecycle

The header depends only on existing stable types:

- `common::Status`
- `common::LogIndex`
- `common::Term`
- `raft::OpaquePayload`

## Apply Input and Return Semantics

`Apply(...)` accepts:

- committed log index
- committed term
- opaque business command payload

Contract notes captured in the header and tests:

- caller is responsible for invoking `Apply(...)` only for committed entries
- caller is responsible for strictly increasing `LogIndex` order
- interface does not implement duplicate suppression in T036
- returned `Status` is explicit success/failure; no `bool`, no exceptions
- failure must not be treated as business success by the caller

This matches the current T023/T024 apply path, where `ApplyWork` already carries:

- `index`
- `term`
- `type`
- `payload`

and the runtime apply callback currently receives a `LogEntry`.

## CreateSnapshot and RestoreSnapshot Boundaries

`CreateSnapshot(...)` and `RestoreSnapshot(...)` are intentionally narrow:

- `CreateSnapshot(...)` produces only the business snapshot payload
- `RestoreSnapshot(...)` consumes only the business snapshot payload

They do **not** take or manage:

- `SnapshotMetadata`
- membership state
- hard state
- log ranges
- storage objects
- runtime objects

This keeps the boundary aligned with the existing Raft snapshot split:

- Raft still owns `last_included_index`
- Raft still owns `last_included_term`
- Raft still owns membership metadata
- storage still owns persistence/checksum/file replacement

## Ownership, Lifecycle, and Threading Convention

The interface does not introduce a factory, singleton, or service locator.

Conventions:

- ownership stays with the composition layer
- lifetime is external to `IRaftStateMachine`
- thread serialization remains the caller’s responsibility
- no internal mutex, worker thread, queue, or scheduler was added

This preserves the existing runtime boundary from T023/T024 instead of changing apply worker behavior.

## MetadataCommand Model

`MetadataCommand` is defined as a small metadata-layer command envelope:

- `MetadataCommandType type`
- `std::string command_id`
- `std::string target_id`
- `std::optional<std::uint64_t> expected_generation`
- `raft::OpaquePayload payload`

The first version intentionally supports one generic type:

- `MetadataCommandType::OPAQUE_OPERATION`

This is enough for T036 because the task requires a deterministic durable envelope, not concrete store/task/placement business commands.

T036 intentionally does **not** define:

- `StoreInfo`
- `StoreState`
- `PlacementRequest`
- `TaskRecord`
- concrete Phase9 command enums

## Command ID, Target, and Generation Semantics

Validation rules implemented in `ValidateMetadataCommand(...)`:

- `type` must be recognized
- `command_id` must not be empty
- `target_id` must not be empty
- `expected_generation`, when present, must be non-zero
- `command_id`, `target_id`, and `payload` must stay within fixed safe size limits

Semantics:

- `command_id` is the durable command identity for future idempotency logic
- `target_id` is a stable opaque target identifier string, without binding T036 to store/task-specific structures
- `expected_generation` is optional optimistic-concurrency metadata for future phases
- empty `payload` is allowed; this keeps the envelope usable for future command kinds that do not need embedded opaque body bytes

## Serialization Format, Version, Byte Order, and Determinism

`EncodeMetadataCommand(...)` and `DecodeMetadataCommand(...)` implement a deterministic binary format.

Version:

- format version = `1`

Byte order:

- all integer fields use explicit big-endian encoding

Current layout:

1. `u8` format version
2. `u8` command type
3. `u16` flags
4. `u32` command_id length
5. `u32` target_id length
6. `u32` payload length
7. optional `u64` expected_generation when the flag is set
8. command_id bytes
9. target_id bytes
10. payload bytes

Determinism rules:

- fixed field order
- explicit integer widths
- explicit endianness
- no struct memory layout reuse
- no `size_t` encoding
- no unordered iteration
- no protobuf/JSON

The same `MetadataCommand` produces identical bytes across supported platforms/compilers.

## Decode Boundary and Error Mapping

Decode is strict and atomic:

- unknown version -> `CORRUPTION`
- unknown flags -> `CORRUPTION`
- unknown type in encoded bytes -> `CORRUPTION`
- truncated header/field/data -> `CORRUPTION`
- oversized encoded lengths -> `CORRUPTION`
- trailing bytes -> `CORRUPTION`
- invalid semantic field content after decode -> `INVALID_ARGUMENT`
- null output pointer -> `INVALID_ARGUMENT`

Implementation details:

- decode uses a bounded cursor reader
- lengths are validated before allocation/use
- decode writes into a temporary `MetadataCommand`
- output object is assigned only after full successful decode

This satisfies the “decode failure must not return partially valid output” requirement.

## Snapshot Payload vs. Raft SnapshotMetadata Responsibility

T036 keeps two separate boundaries:

- `MetadataCommand` encoding is for command payloads stored in normal log entries
- `IRaftStateMachine` snapshot payloads are for business-state snapshot bytes only

It does **not** merge either of them with:

- `raft::SnapshotMetadata`
- Raft membership metadata
- hard state
- persistence checksum/file format
