## MetadataStateMachine Internal State Model

`MetadataStateMachine` owns only deterministic business state:

- `std::map<std::string, MetadataStateRecord> records_`
- `last_applied_index_`
- `last_applied_term_`
- one `std::mutex` for future apply/query cross-thread safety

`MetadataStateRecord` contains:

- `target_id`
- `payload`
- `generation`
- `last_command_id`
- `last_update_index`
- `last_update_term`

The state machine does **not** hold:

- `RaftCore`
- `RaftRuntime`
- `RaftLog`
- `IRaftStorage`
- transport/RPC objects
- store/task/placement business types

## Apply Index Ordering and Duplicate Protection Strategy

`Apply(...)` enforces strict sequential progress:

- `index == 0` is rejected
- `term == 0` is rejected
- first successful apply must use index `1`
- later successful applies must use exactly `last_applied_index + 1`

Duplicate and ordering behavior:

- same index as current `last_applied_index` -> `BUSY`
- smaller-than-expected index -> `BUSY`
- larger-than-expected index (gap) -> `BUSY`

This means T037 uses a strict no-gap policy aligned with the current T023/T024 runtime apply contract.

There is no separate command-id dedupe table in T037. Duplicate protection is based on applied log index only.

## MetadataCommand Apply Semantics

`Apply(...)` decodes the T036 `MetadataCommand` and applies the current minimal deterministic metadata semantics:

- command type remains `OPAQUE_OPERATION`
- `target_id` selects the record to create or overwrite
- `payload` becomes the current stored payload for that target
- `command_id` is stored as the record’s `last_command_id`

Behavior:

- first update for a target creates generation `1`
- later successful updates increment generation by `+1`
- the same command at a different log index is treated as a new update, not as an idempotent replay

This is deliberate scope control: T037 preserves `command_id` for observability and future phases, but does not yet implement command-id replay suppression.

## expected_generation and Generation Progression

Rules:

- no `expected_generation`: unconditional create/overwrite
- with `expected_generation`:
  - target must already exist
  - current generation must match exactly
  - mismatch returns `BUSY`
  - missing target returns `NOT_FOUND`

Generation progression:

- new target -> generation `1`
- existing target success -> `generation + 1`
- any failed apply leaves generation unchanged

This provides a minimal optimistic-concurrency rule without introducing Phase9 business semantics.

## Query Interface and Return Model

`MetadataStateMachine` adds its own read-only query API:

- `GetRecord(const std::string &, MetadataStateRecord *) const`
- `GetLastApplied(LogIndex *, Term *) const`

Query behavior:

- missing target -> `NOT_FOUND`
- empty target id -> `INVALID_ARGUMENT`
- returns value copies only
- does not mutate state
- does not advance apply progress

The returned `MetadataStateRecord` gives the verification data T037 needs:

- payload
- generation
- last command id
- last update index/term

## Atomic Update Strategy

`Apply(...)` follows a validate-then-commit pattern:

1. decode and validate the command
2. validate index sequencing
3. validate generation precondition
4. build candidate record
5. clone the record map
6. update the clone
7. replace visible state and apply progress together

Failure does not advance:

- `records_`
- `last_applied_index_`
- `last_applied_term_`

This keeps partial metadata updates from leaking on any rejection path.

## Business Snapshot Format, Version, Field Order, and Byte Order

The business snapshot format is separate from `MetadataCommand` and from Raft snapshot metadata.

Version:

- snapshot format version = `1`

Byte order:

- all integer fields use explicit big-endian encoding

Current layout:

1. `u8` snapshot format version
2. `u32` record count
3. `u64` last_applied_index
4. `u64` last_applied_term
5. repeated records in sorted `target_id` order:
   - `u32` target_id length
   - `u32` payload length
   - `u32` last_command_id length
   - `u64` generation
   - `u64` last_update_index
   - `u64` last_update_term
   - target_id bytes
   - payload bytes
   - last_command_id bytes

The snapshot payload contains only business state and business progress.

## Snapshot Determinism Strategy

Determinism is guaranteed by:

- `std::map` ordered by `target_id`
- fixed field order
- explicit integer widths
- explicit big-endian encoding
- no raw object serialization
- no unordered iteration
- no JSON/protobuf

Repeated `CreateSnapshot(...)` on the same state produces byte-identical output. This is covered by tests for both empty and populated states.

## RestoreSnapshot Validation and Atomic Replacement

`RestoreSnapshot(...)`:

- decodes through a bounded cursor reader
- validates version, record count, lengths, progress, duplicate keys, generation, and trailing bytes
- constructs a temporary `std::map`
- only swaps visible state after full validation succeeds

Rejected cases include:

- unknown version
- truncated payload
- oversized record count
- oversized string/payload lengths
- duplicate `target_id`
- zero generation
- invalid record progress
- trailing garbage

On failure, current state and progress remain unchanged.

## Snapshot Plus Command Replay Consistency

T037 added explicit replay equivalence testing:

- full command replay from empty state
- snapshot after a prefix
- restore that snapshot into a fresh machine
- replay the remaining commands

The final checks match:

- target payloads
- generations
- last command ids
- last applied index
- last applied term

This demonstrates that business snapshot restore plus later command replay converges to the same result as full replay.

## Threading, Lifecycle, and Ownership Convention

`MetadataStateMachine`:

- owns no threads
- owns no queues
- owns no runtime/storage/network object
- is expected to be owned by a future composition layer

The mutex is used only for protecting state consistency between future `Apply(...)` and `Query(...)` callers. It is not used as a scheduling mechanism.
