## MetadataService Responsibilities and Dependencies

`MetadataService` holds non-owning pointers to:

- `raft::RaftRuntime`
- `MetadataStateMachine`

It does not hold or include:

- `RaftCore`
- `RaftLog`
- `PeerProgress`
- `IRaftStorage`
- Transport
- RPC/protobuf/gRPC generated types

The service owns no worker thread, queue, server, or consensus state. Future composition code owns object lifetime.

## Proposal Request and Result Model

Public proposal entry points:

- `Propose(const MetadataCommand&, MetadataProposalResult*)`
- `Propose(const MetadataCommand&, std::chrono::milliseconds, MetadataProposalResult*)`

`MetadataProposalResult` includes:

- `Status`
- runtime `proposal_id`
- `command_id`
- final applied log index
- leader id/address hint when Runtime knows it
- `final_result`

`proposal_id` is generated per service instance with an atomic counter. `command_id` is preserved from the caller and is not replaced by the service.

## MetadataCommand Validation and Encoding Path

`Propose()` calls:

1. `ValidateMetadataCommand(command)`
2. `EncodeMetadataCommand(command, &payload)`
3. `RaftRuntime::EnqueueProposal(ProposalEvent{proposal_id, payload})`

Validation or encoding failure returns before Runtime enqueue. The service does not duplicate the T036 binary codec and does not inspect Phase9 Store/Task/Placement payloads.

## RaftRuntime Enqueue Method

The service uses only `RaftRuntime::EnqueueProposal()`.

It does not:

- call `RaftCore::Propose()`
- append directly to `RaftLog`
- call `MetadataStateMachine::Apply()` to fake success
- read commit index or quorum state

## Final Success Condition

Success is whatever Runtime returns as the final `ProposalResult`. Runtime emits success only after the log entry is committed and the apply callback succeeds.

T038 also fixed two Runtime result-semantics defects discovered by the new tests:

- ordinary proposal results now carry leader hint fields, reusing existing `ResolveLeaderHint()`
- persistence failure now completes affected pending proposal results instead of leaving callers to time out
- apply scheduling is limited to committed entries whose log index is already stable
- persistence worker appends log entries before saving a HardState whose `commit_index` may reference those same entries

These changes do not alter the Runtime thread model, queue structure, RaftCore algorithm, or storage format.

## Result Correlation and Concurrency

`MetadataService` waits with a bounded deadline and polls `CollectProposalResults()`.

Concurrency isolation:

- one runtime proposal id per request
- one mutex protects cached collected results
- batch collection caches unmatched results for other waiting requests
- abandoned proposal ids prevent late timeout results from completing a later request
- cache and abandoned-id sets are bounded at 4096 entries

The service creates no worker thread. Waiting happens on the caller thread only.

## Timeout, Late Result, and Runtime Stop Semantics

- Timeout returns `RetryLater("timed out waiting for metadata proposal result")`
- Timeout means the caller stopped waiting; the Raft proposal may still later commit/apply
- Late results for timed-out proposal ids are discarded when observed
- Runtime stop while waiting returns `Busy`
- Runtime not running or stopped causes enqueue failure and no service-side wait state remains

No log rollback or cancellation protocol was added.

## NOT_LEADER and Leader Hint Propagation

Non-leader proposal rejection still comes from Runtime/RaftCore.

Because `ProposalResult` now includes:

- `leader_id`
- `leader_address`

`MetadataService` can preserve known leader hints without maintaining a second role/leader truth. Unknown leader remains `kInvalidNodeId` with an empty address.

## Persistence and Apply Failure Propagation

Failures are returned as Runtime final results:

- persistence failure returns the storage error, such as `IoError`
- state-machine apply failure returns the apply status, such as `NotFound` or `Busy`
- business failure does not become service success

The service does not retry, reinterpret, or mask these errors.

## Query Interface and Local Consistency Boundary

Public query entry:

- `Query(const std::string& target_id, MetadataQueryResult*) const`

`MetadataQueryResult` includes:

- value-copy `MetadataStateRecord`
- local `last_applied_index`
- local `last_applied_term`

Query behavior:

- reads only `MetadataStateMachine`
- returns `NotFound` for missing target
- returns `InvalidArgument` for empty target or null output
- does not advance apply progress
- does not read RaftLog
- does not implement ReadIndex, lease-read, follower-read, or linearizable read protocol

## expected_generation Final Decision Path

`MetadataService` performs command structure validation only. It does not pre-read state to decide `expected_generation`.

Generation conflicts are decided by `MetadataStateMachine::Apply()` in committed log order and are returned through the Runtime proposal result.

## command_id Current Semantics

`command_id` is preserved in `MetadataCommand` and echoed in `MetadataProposalResult`.

Current scope:

- command identity and diagnostics only
- no global command-id dedupe table
- no idempotency promise beyond T037 log-index duplicate protection

## MetadataStateMachine and Runtime Apply Binding

Tests bind the same `MetadataStateMachine` instance to Runtime through the existing `RaftRuntime::ApplyFn` callback:

```cpp
return machine.Apply(entry.index, entry.term, entry.payload);
```

`MetadataService` queries that same instance. It never invokes `Apply()` directly during proposal submission.
