## Test Assembly and Runtime Driving

`state_machine_replacement_test.cpp` adds a file-local `RuntimeHarness` that:

- creates a single-node leader `RaftCore`
- uses `MemoryRaftStorage`
- starts `RaftRuntime`
- binds `RaftRuntime::ApplyFn` to an `IRaftStateMachine`
- submits opaque payloads only through `RaftRuntime::EnqueueProposal`
- collects final results only through `RaftRuntime::CollectProposalResults`

No test server, RPC adapter, network listener, fixed port, or external service is used.

## AlternateStateMachine

The test-only `AlternateStateMachine` implements only:

- `Apply`
- `CreateSnapshot`
- `RestoreSnapshot`

It does not inherit `MetadataStateMachine`, does not parse `MetadataCommand`, and has no Store, Task, Placement, RPC, Transport, membership, or RaftCore knowledge. Its snapshot format is a private opaque test format owned by that implementation.

## Verified Areas

- **IRaftStateMachine replacement**: Runtime applies opaque proposals through an alternate implementation without changing Raft consensus, storage, transport, membership, or runtime structure.
- **Proposal persistence/commit/apply chain**: A blocking apply test verifies final proposal success is not emitted before `Apply` completes.
- **Apply order and uniqueness**: Sequential opaque proposals apply at log indexes 1, 2, and 3; duplicate apply completion does not reapply or complete the wrong request.
- **MetadataService end-to-end**: MetadataService submits `MetadataCommand`, Runtime commits/applies it, Query sees payload, generation, command_id, and local applied progress.
- **Concurrent result isolation**: Concurrent MetadataService proposals keep distinct command/proposal results.
- **Error propagation**: Existing target tests cover NOT_LEADER, known/unknown leader hint behavior, queue/runtime stopped behavior, persistence failure, apply failure, timeouts, and late-result isolation.
- **Snapshot and recovery**: MetadataStateMachine business snapshot restore plus incremental replay matches full Runtime-driven replay. AlternateStateMachine independently snapshots/restores its own opaque payload.

## Recovery Boundary

Current public `RaftRuntime::Initialize` does not rebuild `RaftCore` log/apply state from `IRaftStorage::Load`; T039 did not add a production restart/recovery framework. Coverage is therefore:

- covered: business state-machine snapshot restore plus subsequent command replay
- covered: MemoryRaftStorage-backed Runtime proposal/apply path before snapshot
- not covered: full process restart that automatically reloads persisted Raft log/snapshot into Runtime/Core

This is consistent with the T039 boundary that forbids expanding into production recovery infrastructure.
