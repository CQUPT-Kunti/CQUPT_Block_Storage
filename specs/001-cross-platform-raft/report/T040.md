## StoreInfo and StoreState Design

`StoreState` contains exactly:

- `RUNNING`
- `STOPPED`
- `FAILED`

`StoreInfo` fields:

- `StoreId id` (`std::uint64_t`; `0` is invalid, matching existing proto/common id style)
- `StoreAddress address` (`host`, `port`)
- `capacity_bytes`
- `used_bytes`
- `StoreState state`
- `generation`
- `last_heartbeat_ms`

`StoreInfo` does not contain protobuf/gRPC types, Runtime objects, connections, file handles, or threads.

## Persistent vs Leader-Local Fields

Persistent fields:

- `id`
- `address`
- `capacity_bytes`
- `used_bytes`
- `state`
- `generation`

Leader-local transient field:

- `last_heartbeat_ms`

`PersistentStoreInfo` intentionally excludes `last_heartbeat_ms`. `ExportPersistent()` and `RestorePersistent()` round-trip only durable registry state. Heartbeat updates do not advance `generation` and do not change persistent fields.

## StoreRegistry Indexes and Deterministic Sorting

Internal indexes:

- `std::map<StoreId, StoreInfo> stores_`
- `std::map<std::string, StoreId> address_index_`

`std::map` provides deterministic list order by store id. Address uniqueness is enforced through a normalized `host:port` key.

All query/list methods return value copies and do not expose mutable internal references.

## Registration and Duplicate Semantics

`RegisterStore()`:

- rejects id `0`
- rejects empty host
- rejects port `0`
- rejects zero capacity
- rejects `used_bytes > capacity_bytes`
- rejects invalid `StoreState`
- rejects duplicate id
- rejects duplicate address
- initializes new stores as `RUNNING`
- initializes durable `generation` to `1`

Duplicate registration returns `BUSY`; it is not treated as silent idempotent success.

## Updates, Generation, and expected_generation

`UpdateStore()` supports persistent updates for:

- `capacity_bytes`
- `used_bytes`
- `state`

Rules:

- missing store returns `NOT_FOUND`
- empty update returns `INVALID_ARGUMENT`
- `expected_generation`, if supplied, must match exactly
- generation conflict returns `BUSY`
- successful persistent update increments `generation` by `+1`
- heartbeat does not advance generation
- generation never uses system time
- generation overflow is rejected as `BUSY`

## State, Capacity, and Heartbeat Rules

- State updates allow only `RUNNING`, `STOPPED`, and `FAILED`.
- `ListRunningStores()` returns only `RUNNING` stores.
- `capacity_bytes >= used_bytes` is enforced on register, update, and restore.
- Heartbeat update takes an explicit caller-provided millisecond value.
- Negative heartbeat values are rejected.
- Missing store heartbeat returns `NOT_FOUND`.
- Registry does not read system time, scan timeouts, or auto-mark stores failed.

## Delete and Restore Atomicity

`RemoveStore()` removes both id and address indexes. After removal, the same address can be registered again.

`RestorePersistent()` validates and constructs candidate id/address indexes first, then swaps them into the registry. Invalid duplicate ids, duplicate addresses, invalid capacity, invalid state, or invalid generation leave existing registry state unchanged.

All mutating operations follow validate-then-commit behavior. Failures do not leave partial records, stale address indexes, or generation changes.

## Threading, Lifetime, and Ownership

`StoreRegistry` owns only in-memory value state and one `std::mutex`.

It does not:

- create threads
- hold global state
- call `MetadataService`
- call `RaftRuntime`
- encode `MetadataCommand`
- access files, network, RPC, protobuf, or platform APIs

Lifecycle remains external for later metadata composition.
