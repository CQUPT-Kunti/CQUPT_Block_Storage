## PlacementRequest and PlacementResult

`PlacementRequest`:

- `replica_count`
- `required_capacity_bytes`

Rules:

- `replica_count == 0` is rejected as `INVALID_ARGUMENT`
- `required_capacity_bytes == 0` is rejected as `INVALID_ARGUMENT`

Zero required capacity is intentionally not allowed in T041 because placement is for allocating capacity; accepting zero would create ambiguous no-op placement semantics.

`PlacementResult`:

- `selected_store_ids`

Success returns exactly `replica_count` unique store ids in deterministic order. Failures leave the caller-provided output unchanged.

## IPlacementPolicy Interface

`IPlacementPolicy` exposes one responsibility:

- `SelectStores(const PlacementRequest&, const std::vector<StoreInfo>&, PlacementResult*)`

The interface consumes a value snapshot of candidate stores. It does not depend on or hold:

- `StoreRegistry`
- Raft/Core/Runtime
- MetadataService
- RPC/protobuf/gRPC
- storage or files

It does not mutate candidates, reserve capacity, update generation, update heartbeat, or submit consensus work.

## SimplePlacementPolicy Rules

Candidate validation rejects:

- store id `0`
- empty host
- port `0`
- invalid `StoreState`
- generation `0`
- `capacity_bytes < used_bytes`
- duplicate store id
- duplicate address

Selection rules:

- only `StoreState::RUNNING` stores are eligible
- STOPPED and FAILED stores are excluded
- remaining capacity is `capacity_bytes - used_bytes`, computed only after `capacity >= used`
- a store is eligible only when remaining capacity is at least `required_capacity_bytes`
- the same store can appear at most once

## Deterministic Sorting

Eligible stores are sorted by:

1. remaining capacity descending
2. `StoreId` ascending

The policy builds deterministic `std::map`-based candidate validation and uses `std::sort` with an explicit comparator. It does not depend on input order, unordered containers, random selection, time, or process state.

## Error Semantics

- invalid request/candidate snapshot: `INVALID_ARGUMENT`
- not enough RUNNING stores: `RESOURCE_EXHAUSTED` with `not enough running stores for placement`
- enough RUNNING stores but not enough capacity-qualified stores: `RESOURCE_EXHAUSTED` with `not enough store capacity for placement`

No partial selection is returned as success.

## StoreRegistry Snapshot Boundary and Side Effects

`SimplePlacementPolicy` does not lock or hold `StoreRegistry`. Tests pass a value snapshot from `ListStores()` and verify:

- later Registry changes do not change already captured input
- successful placement does not change `used_bytes`, `generation`, or `last_heartbeat_ms`
- placement success does not imply reservation

Concurrent callers may select the same store from the same snapshot; that coordination is intentionally left to later metadata consensus/business layers.

## Replaceability

Tests define a file-local `AscendingPlacementPolicy` implementing `IPlacementPolicy`. It replaces `SimplePlacementPolicy` through the interface without changing StoreRegistry, Raft, MetadataStateMachine, MetadataService, RPC, or CMake production structure.

## Threading and Lifecycle

Lifecycle is external to future metadata composition.
