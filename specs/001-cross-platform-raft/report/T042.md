## Design

- **Task model**: `TaskRecord` contains `task_id`, `TaskType`, `TaskState`, target payload, assigned `StoreId`, result payload, `generation`, and FIFO `sequence`.
- **Task types**: `CREATE`, `DELETE`, `COPY`, `CUSTOM`.
- **Task states**: `WAITING`, `RUNNING`, `SUCCESS`, `FAILED`.
- **Payloads**: `std::vector<common::Byte>` opaque bytes. No protobuf, gRPC, Raft or MetadataCommand types are exposed.
- **TaskManager state**: one mutex-protected `std::map<TaskId, TaskRecord>` plus `next_sequence_`. Lists are sorted by `sequence`, then `task_id`.
- **Create semantics**: valid new task starts `WAITING`, `assigned_store = 0`, `generation = 1`; duplicate same persistent content is idempotent success; duplicate conflicting content returns `BUSY`.
- **Assignment semantics**: `AssignTask` and `AssignNextWaitingTasks` move `WAITING -> RUNNING`, set the store, and increment generation. Same running task for the same store is idempotent.
- **Result semantics**: `CompleteTask` accepts only `SUCCESS` or `FAILED` from the assigned store; duplicate same final report is idempotent; conflicting final reports are rejected.
- **Generation rules**: generation starts at 1 and advances on persistent state transitions. Optional `expected_generation` must match before assignment or completion.
- **Persistent view**: `ExportPersistent` returns deterministic records; `RestorePersistent` validates all records first, then atomically replaces state. No file format, Raft proposal, or MetadataCommand encoding is introduced.

## Dispatcher

- **ITaskDispatcher**: replaceable boundary with only `PollTasks` and `ReportTaskResult`.
- **SimpleTaskDispatcher**: thin wrapper over `TaskManager`; polling uses `AssignNextWaitingTasks`, applies a dispatcher maximum poll cap, and never returns already running or terminal tasks.
- **Poll atomicity**: assignment happens under `TaskManager` lock, so concurrent pollers cannot receive the same task.
- **Side effects**: polling only moves selected tasks to `RUNNING`; reporting only completes assigned tasks. No capacity reservation, retry queue, timeout scanner or Store RPC was implemented.
