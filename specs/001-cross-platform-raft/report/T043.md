## Design

- **RPC Adapter**: `StoreRpcServiceAdapter` implements all six existing `store.proto` RPCs: `Register`, `Heartbeat`, `Stop`, `Remove`, `PollTasks`, and `ReportTaskResult`.
- **Service split**: RPC layer converts protobuf to internal store/task types, calls `MetadataService`, maps business `Status` to `RpcStatus`, and returns `grpc::Status::OK` for business rejection.
- **Persistent truth**: `MetadataStateMachine` owns the applied `StoreRegistry` and `TaskManager`. RPC and service code do not directly mutate persistent store/task state.
- **Persistent command path**: Register, Stop, Remove, PollTasks and ReportTaskResult build `MetadataCommandType::STORE_OPERATION`, submit through `MetadataService::Propose`, wait for committed Apply, then decode the applied result record.
- **Heartbeat path**: Heartbeat calls `MetadataStateMachine::UpdateStoreHeartbeat` through `MetadataService`; it does not create `MetadataCommand`, does not enqueue Raft proposal, does not advance persistent generation, and does not write Raft log.

## Store Business Commands

- Added `MetadataCommandType::STORE_OPERATION` while keeping the existing T036 outer command format and version.
- Added metadata-layer `StoreBusinessCommand` and `StoreBusinessResult` with deterministic v1 binary encoding.
- Encoding uses fixed big-endian integers, fixed field order, explicit version byte, strict unknown-version rejection, strict truncation rejection, and trailing-byte rejection.
- Store business commands cover:
  - `REGISTER_STORE`
  - `UPDATE_STORE_STATE`
  - `REMOVE_STORE`
  - `CREATE_TASK` for test/business setup, not exposed as Store RPC
  - `POLL_TASKS`
  - `REPORT_TASK_RESULT`

## StateMachine Apply

- Existing `OPAQUE_OPERATION` behavior is preserved for T036-T039.
- `STORE_OPERATION` decodes the internal store command and applies it in log index order.
- Store registration applies through `StoreRegistry::RegisterStore`.
- Stop applies through `StoreRegistry::UpdateStore` with `STOPPED` and expected generation.
- Remove checks expected generation and applies through `StoreRegistry::RemoveStore`.
- PollTasks applies through `TaskManager::AssignNextWaitingTasks`, so task selection and `WAITING -> RUNNING` are persisted.
- ReportTaskResult applies through `TaskManager::CompleteTask`, preserving duplicate result idempotency.
- Applied store command results are stored in the command target metadata record so `MetadataService` can read the exact applied result after proposal completion.

## RPC Mapping

- `Register`: validates request id, proposes persistent register command, returns applied `StoreDescriptor`.
- `Heartbeat`: updates leader-local heartbeat marker only; current proto has no heartbeat timestamp field, so the adapter uses request `generation` as the explicit caller-provided heartbeat marker.
- `Stop`: proposes persistent `STOPPED` state transition with expected generation.
- `Remove`: proposes persistent remove with expected generation.
- `PollTasks`: proposes persistent task assignment and returns applied task assignments.
- `ReportTaskResult`: proposes persistent final task result and returns duplicate-result information.
- `NOT_LEADER`: write proposals that fail before log index assignment with leader hint are mapped to `RPC_STATUS_CODE_NOT_LEADER`.
- Business errors such as duplicate address, invalid capacity, generation conflict, missing task and conflicting result stay in `RpcStatus` with gRPC OK.

## Boundaries

- Store RPC does not start a gRPC server, bind a port, create client channels, or manage lifecycle.
- RPC layer does not call RaftCore, RaftLog, HardState, commit index, PeerProgress or quorum logic.
- No proto files were changed.
- No Raft runtime thread, queue, persistence gate or proposal completion semantics were changed.
- No automatic heartbeat timeout scan, automatic FAILED conversion, retry, requeue, capacity reservation, migration, ReadIndex, lease-read, client SDK, Phase10 fixture/tooling or production server composition was implemented.
