## Test Harness and Runtime Drive

- The new `StoreMetadataTest` fixture builds a single-node deterministic `RaftCore` leader or follower, `MemoryRaftStorage`, `RaftRuntime`, `MetadataStateMachine`, `MetadataService`, and `StoreRpcServiceAdapter`.
- Store control calls use direct gRPC adapter method calls with `grpc::ServerContext`; no production `Server`, listener, fixed port, or external network is created.
- Runtime Apply uses the existing `RaftRuntime::ApplyFn` callback and applies only `LogEntryType::COMMAND` payloads into the same `MetadataStateMachine` instance used by queries.
- A file-local `BlockingApply` helper verifies proposal success is gated on committed Apply completion without introducing a general workflow or cluster framework.

## Store Lifecycle Coverage

- Valid Store registration is verified through Store RPC -> MetadataService -> Runtime proposal -> committed Apply -> MetadataStateMachine query.
- Duplicate Store ID and invalid capacity/used combinations are rejected and do not create new records.
- STOPPED state is applied through the Store RPC stop path and generation advances only after the consensus-applied update.
- FAILED state is covered through a direct `MetadataStateMachine` Store command apply path because T043 Store RPC currently exposes Stop/Remove but no separate Fail RPC.
- RemoveStore is verified to remove the record through the applied Metadata command path.

## Heartbeat Boundary

- Heartbeat updates `last_heartbeat_ms` through the MetadataService local state-machine query boundary.
- Heartbeat does not advance `last_applied_index`, persistent generation, or `used_bytes`.
- A snapshot/restore round trip verifies `last_heartbeat_ms` is not persisted.
- The current Store RPC heartbeat adapter does not perform a non-Leader rejection because existing allowed runtime boundaries do not expose a role query for that call; persistent Store write RPCs still return `NOT_LEADER` through the proposal path.

## Placement Coverage

- Placement consumes a value snapshot from `MetadataStateMachine::ListStores`.
- `SimplePlacementPolicy` excludes STOPPED, FAILED, and insufficient-capacity stores.
- Selection is verified against applied Store state and does not mutate Store generation, heartbeat, or capacity.
- Failed placement leaves an existing output object unchanged.

## Task Lifecycle Coverage

- Task creation uses the existing `MetadataService::CreateTaskForTest` internal test/business entry.
- PollTasks is verified through Store RPC and applies task assignment through Metadata consensus.
- Duplicate polling is covered by verifying already RUNNING or terminal tasks are not reselected and by concurrent poll calls that produce no duplicate task IDs.
- ReportTaskResult is verified through Store RPC, including duplicate result idempotency.

## Proposal Gate and Error Propagation

- Blocking Apply proves Store registration does not return success before MetadataStateMachine Apply completes.
- Non-Leader Store registration returns a normal business response with `NOT_LEADER` and known leader hint.
- Runtime stopped rejects new Store proposal requests with a clear busy status.
- Persistence failure from `IRaftStorage::AppendEntries` propagates as `IO_ERROR` and does not apply the Store.
- Apply failure propagates as `INVALID_ARGUMENT` and leaves the Store absent.

## Snapshot and Recovery Coverage

- Version 2 MetadataStateMachine snapshot restores Store and Task business state.
- Snapshot plus incremental command replay is compared with full replay for equivalent Store and Task state.
- Snapshot restore rejects trailing corrupt bytes and preserves existing state.
- Legacy version 1 empty snapshot remains accepted.
- Coverage is limited to business snapshot payload and incremental replay; full process restart and RaftStorage replay remain outside T044.
