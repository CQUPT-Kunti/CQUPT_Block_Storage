## Fixture Format and Version Source

No existing fixture/testdata files were present in the repository. `plan.md` and `data-model.md` require versioned fixtures with scenario name, checksum/verification metadata, input records/messages/config, and expected results, but do not define a concrete schema.

T045 therefore uses a minimal line format for test fixtures:

```text
version: 1
scenario: basic-election
node: 1,node-1,leader,127.0.0.1,9001
step: 1,tick,1
expect: leader,1
checksum: <semantic checksum>
```

Rules:

- `version` is required and must be `1`.
- `scenario` is required.
- `checksum` is required and validates canonical semantic lines excluding comments, blank lines, and the checksum line.
- `node`, `step`, and `expect` are repeatable.
- Unknown fields are rejected.

## FixtureLoader Public Interface

```cpp
class FixtureLoader {
public:
    static common::ChecksumValue ComputeChecksumForText(std::string_view text);
    static common::Status LoadFromString(std::string_view text, Fixture *fixture);
    static common::Status LoadFromFile(const std::filesystem::path &path,
                                       Fixture *fixture);
};
```

The loader returns ordinary value objects:

- `Fixture`
- `FixtureNode`
- `FixtureStep`
- `FixtureExpectation`

No returned value contains references into parser buffers.

## Loaded Value Model

- `Fixture.version`
- `Fixture.scenario`
- `Fixture.checksum`
- ordered `Fixture.nodes`
- ordered `Fixture.steps`
- ordered `Fixture.expectations`

Node fields:

- `node_id`
- `name`
- `role`: follower, candidate, leader, learner
- `address.host`
- `address.port`

Step fields:

- `step_id`
- `type`: tick, propose, deliver, partition, heal
- `node_id`
- optional `target_node_id`
- optional payload string

## File Reading and Path Strategy

- `LoadFromFile` uses `std::filesystem` and `std::ifstream`.
- It reads only the caller-provided file.
- It rejects missing paths, directories, non-regular paths, and files above the fixed size limit.
- It does not scan directories, expand globs, follow fixture includes, read environment variables, or assume the current working directory is the repository root.
- Tests use `std::filesystem::temp_directory_path()` and clean their temporary directory via RAII.

## Parsing and Validation Rules

Syntax validation:

- Rejects empty input.
- Rejects lines without `:`.
- Rejects duplicate `version`, `scenario`, and `checksum`.
- Rejects unknown top-level fields.
- Rejects bad CSV field counts for `node`, `step`, and `expect`.

Field validation:

- Rejects unsupported fixture versions.
- Rejects missing required fields.
- Rejects unsigned integer negative values and parse failures.
- Rejects node id `0`.
- Rejects empty node/scenario/expectation strings.
- Rejects invalid roles and step types.
- Rejects invalid ports outside `1..65535`.

Cross-field validation:

- Requires at least one node.
- Rejects duplicate node ids.
- Rejects duplicate node names.
- Rejects duplicate node addresses.
- Rejects more than one initial leader.
- Rejects duplicate step ids.
- Rejects step references to unknown nodes.
- Requires target node for `deliver`, `partition`, and `heal`.

## Unknown Version and Unknown Field Handling

- Unknown version: rejected with `InvalidArgument`.
- Unknown field: rejected with `InvalidArgument`.
- No forward-compatible silent ignore behavior is implemented.

## Resource Limits

Fixed limits:

- fixture text/file size: 64 KiB
- nodes: 64
- steps: 1024
- expectations: 1024
- individual strings: 256 bytes

Limit violations return `ResourceExhausted`.

## Error Model and Output Atomicity

- Null output: `InvalidArgument`.
- Missing file: `NotFound`.
- File inspect/open/read failure: `IoError`.
- Syntax/field/cross-field errors: `InvalidArgument`.
- Checksum mismatch: `Corruption`.
- Resource limits: `ResourceExhausted`.

`LoadFromString` and `LoadFromFile` build a candidate `Fixture` first and assign to the caller output only after parsing, checksum validation, and semantic validation all succeed.

## Determinism and Ordering

- Loader does not read system time, random numbers, environment variables, machine info, network, or global mutable state.
- Nodes, steps, and expectations preserve input order.
- Checksum uses normalized semantic lines so comments and blank lines do not affect verification.
- Duplicate detection uses ordered `std::set`, not unordered iteration.
