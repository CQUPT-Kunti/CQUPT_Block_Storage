## Fixture Format and Version Source

No existing fixture/testdata files were present in the repository. The plan and data model require versioned fixtures with scenario name and checksum metadata but do not define a concrete schema.

T045 uses a minimal version 1 line format:

```text
version: 1
scenario: basic-election
node: 1,node-1,leader,127.0.0.1,9001
step: 1,tick,1
expect: leader,1
checksum: <semantic checksum>
```

Checksum is computed over normalized semantic lines, excluding comments, blank lines, and the checksum line.

## FixtureLoader Interface and Value Model

Public API:

```cpp
class FixtureLoader {
public:
    static common::ChecksumValue ComputeChecksumForText(std::string_view text);
    static common::Status LoadFromString(std::string_view text, Fixture *fixture);
    static common::Status LoadFromFile(const std::filesystem::path &path,
                                       Fixture *fixture);
};
```

Loaded values:

- `Fixture`: version, scenario, checksum, ordered nodes, ordered steps, ordered expectations
- `FixtureNode`: node id, name, role, address
- `FixtureStep`: step id, type, source node, optional target node, optional payload
- `FixtureExpectation`: key/value pair

The loader returns ordinary values only; no output contains references into parser buffers.

## FixtureLoader Validation

Syntax and field validation:

- rejects empty input
- rejects malformed `key: value` lines
- rejects duplicate singleton fields
- rejects unknown fields
- rejects unsupported versions
- rejects missing `version`, `scenario`, or `checksum`
- rejects invalid unsigned integers, negative values, invalid ports, empty identifiers, invalid roles, and invalid step types

Cross-field validation:

- requires at least one node
- rejects duplicate node ids, names, and addresses
- rejects multiple initial leaders
- rejects duplicate step ids
- rejects unknown node references
- requires a target node for `deliver`, `partition`, and `heal`

Resource limits:

- fixture text/file size: 64 KiB
- nodes: 64
- steps: 1024
- expectations: 1024
- string field length: 256 bytes

Error mapping:

- missing file: `NotFound`
- read/inspect failure: `IoError`
- syntax/field/cross-field failure: `InvalidArgument`
- checksum mismatch: `Corruption`
- resource limit: `ResourceExhausted`

Output is atomic: failed loads do not modify an existing output fixture.

## File and Path Strategy

- Uses `std::filesystem` and `std::ifstream`.
- Reads only the caller-provided file.
- Does not scan directories, expand globs, load includes, read environment variables, or assume the current working directory.
- Tests use temporary directories via `std::filesystem` and clean them with RAII.

## SimulatedCluster Design

T045 adds a deterministic test-only `SimulatedCluster`:

- builds simulated nodes from a loaded `Fixture`
- preserves deterministic node ordering by node id
- tracks node state as `RUNNING`, `STOPPED`, or `FAILED`
- tracks undirected network partitions between declared nodes
- stores delivered messages in per-node inboxes with monotonic sequence numbers
- provides snapshots of node state and inbox size

It does not create `RaftCore`, `RaftRuntime`, RPC servers, sockets, threads, timers, files, or production Store/Metadata objects.

## Simulated Transport

`SimulatedTransport` implements the existing `transport::IRaftTransport` interface in test support only:

- validates local node and peer ids against `SimulatedCluster`
- supports `Initialize`, `Start`, `Stop`, and `Send`
- rejects sends before start
- rejects unknown peers
- rejects non-running endpoints
- rejects partitioned links with `RetryLater`
- delivers accepted messages to the target node inbox
- returns deterministic canned responses for request messages without running Raft consensus logic

Deterministic response behavior:

- `RequestVoteRequest` -> `RequestVoteResponse` with same term and `vote_granted=true`
- `AppendEntriesRequest` -> `AppendEntriesResponse` with same term, success, and deterministic match index
- `InstallSnapshotRequest` -> `InstallSnapshotResponse` with same term and snapshot boundary

This is a transport harness, not a Raft algorithm executor.
