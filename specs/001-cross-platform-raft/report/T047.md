## client Commands

Supported Metadata commands:

- `metadata-propose`
- `metadata-query`
- `metadata-get-leader`
- `metadata-get-status`

Supported Store commands:

- `store-register`
- `store-heartbeat`
- `store-stop`
- `store-remove`
- `store-poll-tasks`
- `store-report-task-result`

Implementation notes:

- Uses generated gRPC stubs only: `cpr::metadata::v1::MetadataService::Stub` and `cpr::store::v1::StoreControlService::Stub`
- All RPC calls set a bounded deadline through `grpc::ClientContext`
- Parses numeric values strictly with `std::from_chars`
- Rejects negative values before converting to unsigned fields
- Supports `--payload` or `--payload-file` for Metadata propose/query payloads
- Distinguishes transport failure (`grpc::Status`) from business failure (`RpcStatus`)
- Prints `leader_id` and `leader_address` when present in responses

Current consistency boundary stays aligned with the existing proto/service contract:

- `metadata-propose` reports accepted applied result only as returned by the service
- `metadata-query` reports payload/term only and does not claim stronger consistency than the service contract exposes

## raftctl Commands

Supported commands:

- `leader`
- `status`
- `add-learner`
- `promote-learner`
- `remove-member`

Safety handling:

- `remove-member` requires explicit `--yes`
- node id must be positive
- add-learner requires explicit host and port
- the CLI does not compute quorum, catch-up state, or joint consensus rules locally
- membership failures are printed exactly from the service response
- no automatic retry, no forced override, no local bootstrap editing

`raftctl` uses only the public `MetadataService` gRPC contract.

## wal_dump Inputs and Read-Only Parsing

Supported input layouts:

- storage directory containing `hard_state.bin` and `log.bin`, plus optional `snapshot.bin`
- individual record file such as `hard_state.bin` or `log.bin`
- explicit snapshot file with `--snapshot`

Supported visible record data:

- record `format_version`
- `record_type`
- record ordinal and file offset
- `index`
- `term`
- `payload_length`
- checksum `OK` / mismatch
- HardState fields: `current_term`, `voted_for`, `commit_index`, `membership_configuration_id`
- LogEntry fields: `log_entry_type`, payload length
- Snapshot fields: `last_included_index`, `last_included_term`, `membership_configuration_id`, transition flag, voter/learner counts, payload length

Read-only guarantee:

- does not call `FileRaftStorage::Open()`
- does not create files or directories
- does not repair incomplete tails
- does not truncate, rewrite, or replace any input file
- uses explicit file reads and local decoding only

Resource and corruption handling:

- max input file size: 128 MiB
- max record payload length: 64 MiB
- rejects unknown record version
- rejects unknown record type
- rejects truncated header or payload
- rejects checksum mismatch
- rejects oversized payload length before allocation
- rejects malformed snapshot payload through existing `DecodeSnapshotData`

## CLI Parsing, Output, and Exit Codes

- `--help` for all three tools exits `0`
- normal output goes to `stdout`
- errors go to `stderr`
- unknown command/argument, missing required args, invalid numeric input: exit `2`
- gRPC transport/connect failure: exit `3`
- business rejection from response `RpcStatus`: exit `4`
- WAL/snapshot parse or corruption failure: exit `5`

## Deviations, Blockers, or Unfinished

- The current repository still does not have a server-side adapter for Metadata `Propose`, `Query`, `GetLeader`, and `GetStatus`; T047 therefore implements the client side strictly from proto-generated stubs and validates request/response handling with loopback fake gRPC services rather than changing RPC production code.
- `wal_dump` prints lengths and type summaries for opaque payloads; it intentionally does not decode Metadata or Store business payload internals.
