## Server Composition

`ServerApplication` now composes:

- `FileRaftStorage`
- `RaftCore`
- `RaftRuntime`
- `GrpcRaftTransport`
- `MetadataStateMachine`
- `MetadataService`
- `RaftRpcServiceAdapter`
- `MetadataRpcServiceAdapter`
- `StoreRpcServiceAdapter`
- one `grpc::Server`

The server owns lifecycle only. It does not add a generic framework, process
supervisor, or connection-management subsystem.

## Lifecycle Design

Initialization performs:

1. config validation
2. bootstrap member parsing
3. storage open and load
4. `RaftCore` creation
5. `RaftRuntime` creation
6. transport initialization
7. metadata service and RPC adapter creation

Start performs:

1. lifecycle start
2. transport start
3. runtime start
4. gRPC server bind and registration
5. tick thread start
6. peer pump thread start

Stop performs:

1. shutdown request
2. gRPC server shutdown and wait
3. helper thread join
4. runtime shutdown request and wait
5. transport stop
6. lifecycle stop

Repeated `Start()` is safe and returns success when already running.
Repeated `Stop()` is safe.

## Metadata and Store Service Wiring

The production server now registers:

- `raft.v1.RaftService`
- `metadata.v1.MetadataService`
- `store.v1.StoreService`

To make the server composition usable with existing tools and tests, T048 also
completed the missing production metadata RPC methods:

- `Propose`
- `Query`
- `GetLeader`
- `GetStatus`

This required a small public runtime snapshot boundary:

- `RaftRuntime::GetStatusSnapshot(...)`

That boundary is read-only and does not change the runtime thread model.

## Node ID Mapping

Config still stores node IDs as strings such as `node-1`.
Server composition now performs the minimal server-local mapping:

- accept a positive decimal string such as `1`
- or accept `node-N` where `N` is a positive decimal integer

The mapping is kept inside `server.cpp` and is not pushed into raft, transport,
or proto contracts.

## Single-Node Bootstrap Behavior

The existing runtime election path does not self-promote a single-node candidate.
To keep T048 minimal and avoid changing Raft election semantics, the server
bootstrap path promotes a fresh single-voter bootstrap node to leader during
initialization by reusing existing `RaftCore` transitions.

That behavior is limited to:

- one voter
- the local node is that voter
- no learners
- empty persisted state

## First-Version Limitations

The README and implementation now explicitly document these v1 exclusions:

- no startup recovery from non-empty persisted Raft state
- no TLS/auth/authz
- no transport retry/pooling/backoff
- no advanced metadata read consistency modes
- no production orchestration/deployment packaging

If storage load returns non-empty persisted state, initialization fails fast with
a clear status instead of pretending recovery exists.
